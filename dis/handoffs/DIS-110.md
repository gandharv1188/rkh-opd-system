## 1. What ships

An end-to-end test that wires `createServer({ rateLimit, routes.ingest })`
with an injectable clock, bursts four `POST /ingest` requests from the
same `X-Operator-Id`, and asserts the fourth is rejected with:

- HTTP `429`
- `Retry-After` response header set to a positive integer
- Error envelope `{ error: { code: 'RATE_LIMITED', ... } }` from
  DIS-101's global handler

This is the first test that exercises the full request path
(correlation-id → rate-limit → route → error-handler) through
`createServer()`, confirming the DIS-102 middleware is wired into the
composition root and that DIS-101 preserves the `Retry-After` header
set by the middleware.

## 2. Why clock injection

The suite uses `maxTokens: 3` and a synthetic `now: () => clock` so no
real sleeps are needed. Burst is 3 successful calls + 1 rejected call in
<30 ms wall time. Consistent with DIS-102's fake-clock approach, so zero
flakiness risk.

## 3. Test

```
tests/e2e/rate-limit.test.ts
  ✓ Retry-After header present on 429
```

Fake orchestrator implements only `ingest()` (matching `ingest.test.ts`
pattern) and returns distinct `extraction_id` values so the route can
issue 201s for each of the 3 passes before the bucket is exhausted.

## 5. Assumptions

- `createServer()` composes middleware in documented order
  (correlation-id → kill-switch → rate-limit → routes → onError) —
  confirmed by reading `src/http/server.ts`.
- `/ingest` accepts `application/pdf` bodies with `X-Patient-Id`,
  `Idempotency-Key`, and now `X-Operator-Id` headers; fake orchestrator
  returns 201 with `extraction_id`.
- DIS-101 `errorHandler` forwards the `Retry-After` header set by the
  middleware via `c.header(...)` before throw — verified by the 429 body
  having `error.code === 'RATE_LIMITED'` and header present.
