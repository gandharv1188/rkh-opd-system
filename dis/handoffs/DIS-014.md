## 1. What was built

- `dis/src/http/middleware/idempotency.ts` — Hono middleware factory
  `createIdempotencyMiddleware(logger?)`. On state-changing methods
  (POST/PUT/PATCH/DELETE), requires the `Idempotency-Key` header and returns
  400 with `{code: "IDEMPOTENCY_KEY_REQUIRED", message}` when absent or
  whitespace-only. On GET/HEAD/OPTIONS, passes through untouched. The
  resolved stub `IdempotencyResolution = {key, cached:false}` is published on
  the Hono context under `idempotencyKey` for downstream handlers.
- `dis/tests/unit/idempotency.test.ts` — 6 tests driven by
  `app.fetch(new Request(...))`: GET pass-through, POST missing header → 400,
  POST empty header → 400, POST with header → 200 + resolution echoed,
  DELETE enforcement, logger injection + warn/info recording via a spy.

## 3. Decisions

### D-1: `createIdempotencyMiddleware(logger?)` factory (not a bare middleware)

**Why:** DIS-008 (pino logger) is not on this branch. Per the task brief
option (b), expose a factory that accepts an optional logger so a test spy
can be injected and the real pino logger can be wired post-DIS-008 without
touching the middleware's call sites more than once.
**How to apply:** the integrator who lands DIS-008 should import
`createIdempotencyMiddleware(logger)` from the server composition root with
the real pino instance; no change needed in this file.

### D-2: Method-based gating rather than route-list

**Why:** DIS-014's brief allowlists specific state-changing routes
(`POST /ingest`, `POST /extractions/:id/approve`, etc.), but encoding that
list here would duplicate the router table. Gating by HTTP method keeps the
rule mechanical (POST/PUT/PATCH/DELETE → require header; GET/HEAD/OPTIONS →
pass) and matches REST convention. Routes that deliberately want to skip
idempotency can mount a narrower middleware on their own subrouter when
DIS-025 lands.
**Revisit if:** DIS-025 needs per-route idempotency-TTL tuning.

### D-3: Stub resolution is `{key, cached:false}` — no persistent store

**Why:** DIS-025 owns the persistent idempotency store. Returning a
`cached:false` discriminator today keeps the downstream shape stable so
DIS-025 can flip it to `true` on replay without a breaking change.
**How to apply:** DIS-025 should widen `IdempotencyResolution` into a
discriminated union (`{cached:true, storedResponse}` etc.) rather than a
mutable bool.
