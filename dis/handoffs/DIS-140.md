# DIS-140 — UI build + deploy pipeline

## 1. Ticket
DIS-140: Add GitHub Actions workflow to build `dis/ui`, upload artefact, and stage deploy behind the INTEGRATION APPROVED gate.

## 2. Scope
- `.github/workflows/dis-ui-build.yml` — new workflow (build + deploy-staging jobs).
- `dis/ui/scripts/deploy.mjs` — new deploy stub, callable locally and by CI.
- `dis/handoffs/DIS-140.md` — this handoff.

No edits outside these three paths.

## 3. Design
Two-job pipeline:
1. **build** — runs on every push/PR touching `dis/ui/**` or the workflow itself. Installs via `npm ci`, runs `npm run typecheck`, `npm run build`, uploads `dis/ui/dist` as the `dis-ui-dist` artefact (30-day retention).
2. **deploy-staging** — depends on `build`. Runs only when (a) the push is to `main` AND (b) the repo variable `INTEGRATION_APPROVED == 'true'`. Downloads the artefact and runs the stub; real upload is wired once INTEGRATION APPROVED is granted per `dis/document_ingestion_service/06_rollout/rollout.md §6b`.

Paths filter uses explicit globs (`dis/ui/**` and the workflow file) — satisfies VERIFY-1 (non-empty `paths:` list).

## 4. Gate — INTEGRATION APPROVED
The deploy-staging job's `if:` expression requires `vars.INTEGRATION_APPROVED == 'true'`. Until a repo admin sets that variable to `true`, no deploy step runs, even on `main`. This is the explicit rollout §6b guard: artefacts build and are available for inspection, but no staging upload occurs without the human approval signal. Flipping the variable to `false` (or removing it) instantly disables the deploy path without touching code.

Secondary guard: the job is pinned to the `staging` GitHub Environment, so required reviewers / environment protection rules add an additional approval layer if configured.

- Deploy stub is intentionally inert. If a reviewer flips `INTEGRATION_APPROVED` before the real upload is wired, the job will succeed with only a log line — acceptable because no external system is touched.
