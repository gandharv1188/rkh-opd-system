## 1. What ships

Row-level security enabled on `ocr_extractions`, `ocr_audit_log`, `dis_jobs`,
`dis_cost_ledger`. Policies use the generic
`current_setting('app.role', true)` + `current_setting('app.user_id', true)`
pattern so the exact same SQL runs on Supabase and AWS RDS
(portability.md §Database portability). The wiring layer
(`supabase-postgres.ts → setSessionVars`) is already set up to populate
these from JWT claims or service identity.

Policies:

| Table              | SELECT                                    | INSERT       | UPDATE               | DELETE |
| ------------------ | ----------------------------------------- | ------------ | -------------------- | ------ |
| ocr_extractions    | service/admin/nurse, or patient self      | service only | service/nurse/admin  | ∅      |
| ocr_audit_log      | service/admin/nurse                       | service only | blocked by trigger   | ∅      |
| dis_jobs           | service only                              | service only | service only         | service |
| dis_cost_ledger    | service/admin                             | service only | blocked by trigger   | ∅      |

No DELETE policies on DIS clinical tables — CS-2 (rows are never deleted).
Pre-existing tables (`lab_results`, `vaccinations`) retain their existing
`anon_full_access` POC policies; this migration does not touch them.

## 2. Decisions

- **`dis_jobs` allows `FOR ALL`** including DELETE because the queue
  cleanup job (30-day purge) needs to delete completed rows. This is the
  one DIS table where delete is legitimate.
- **Nurse read on ocr_audit_log.** data_model.md's example did not include
  nurse; added so the nurse UI can show "who approved this, when" without
  an admin escalation.
- **`current_setting(..., true)` (missing_ok=true)** on every call so an
  unconfigured session evaluates to empty-string/role='none' rather than
  raising. Defence for test harnesses and the health endpoint that runs
  without session vars.

## 3. Test plan

- Static parse — asserts RLS enabled on all four tables, policy names,
  `current_setting('app.role')` present, no DELETE policy on extractions
  or audit.
- Live (Wave 7): apply → connect as anon (no `app.role` set) → SELECT on
  ocr_extractions → expect empty result (not error). Set app.role='service'
  → expect visibility.

## 4. Dependencies

- M-001, M-002, M-004, M-005 (tables must exist before RLS can be enabled).

## 5. Gotchas

- RLS is bypassed for table owners and for roles with BYPASSRLS. Supabase
  service_role has BYPASSRLS — so service-side work ignores these policies.
  That is intentional; the policies protect anon/authenticated roles.
- `current_setting(..., true)` returns the string `''` when unset, not
  NULL. The `IN ('service', ...)` checks therefore evaluate to false
  (correct — unset role has no access).
