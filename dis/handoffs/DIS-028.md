## 3. Decisions taken during implementation

### D-1: Overloaded `withCorrelation` signature

**Context:** the brief spec was `fn: () => T | Promise<T>` returning `T | Promise<T>`. That is ergonomic but loses the sync-vs-async return type in consumer code.
**Decision:** keep the permissive runtime but add two call signatures so `const n: number = withCorrelation(id, () => 42)` doesn't require a cast while `await withCorrelation(id, async () => x)` still works.
**Revisit if:** a caller ever needs to dispatch based on a dynamically-decided sync/async callback (no current callsite does).

### D-2: Single `Store` type instead of storing the bare string

**Context:** ALS payload could be `string` or `{correlationId}`. The object wrapper lets future tickets extend the scope (request_id, user_id) without a second ALS instance.
**Decision:** `AsyncLocalStorage<{correlationId: string}>`.
**Revisit if:** we want zero-allocation scopes (unmeasured — premature).

### D-3: Process-wide singleton ALS

**Context:** ALS is cheap and intended to be module-scoped.
**Decision:** `const als = new AsyncLocalStorage<Store>()` at module top. All exports share it.
**Revisit if:** we ever need multiple isolated scope trees (not a current requirement).

## 4. What was deliberately NOT done

- No HTTP middleware wiring — DIS-008 already owns the request-side binding (the existing `correlation-id.ts` middleware). A follow-up will wrap `await next()` in `withCorrelation(c.get('correlationId'), ...)` so ALS picks up what the middleware computes. That wiring is outside DIS-028's files_allowed.
- No logger/metrics integration — separate tickets own those sinks; they can now call `currentCorrelationId()` as needed.
- No adoption of the `AsyncLocalStorage.snapshot()` / `bind` APIs (Node ≥ 18) — not required for the current contract.
