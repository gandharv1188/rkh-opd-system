## 3. Decisions taken during implementation

### D-1: Narrow wrapper around pino instead of re-exporting `pino` directly

**Context:** call sites must not forget service/version base fields (§14).
**Decision:** `createLogger()` pre-seeds `base: { service, version }`; the exported `Logger` type is pino's but the construction path is centralised.
**Revisit if:** we swap pino for bunyan/otel-bridge; the type would change.

### D-2: Deferred AppVariables widening for `logger`

**Context:** `src/http/server.ts` is NOT in DIS-008 `files_allowed`; widening `AppVariables` there would leak scope.
**Decision:** middleware casts `c.set('logger', ...)` locally with `(c.set as (k: string, v: unknown) => void)`. The context slot is populated; type coverage arrives in a follow-up ticket.
**Revisit if:** a downstream ticket (e.g. DIS-009 metrics wiring, request-logging middleware) requires typed `c.get('logger')` at the call site.

### D-3: Lazy `getRootLogger()`

**Context:** importing logger must not eagerly write to stdout in unit tests.
**Decision:** module-level cache populated on first call; test factories pass an explicit `destination` stream.

### D-4: No request/response log line yet

**Context:** observability is incremental; TDD §14 calls for correlation id propagation in v1 and detailed access logs later.
**Decision:** bind the child logger now; write access-log middleware in a follow-up (and pair with metrics timing).
**Revisit if:** ops ask for access logs before the follow-up lands.

## 10. Non-obvious gotchas

- Pino emits ISO-string `time` only when `timestamp: pino.stdTimeFunctions.isoTime` is set; default is numeric ms-since-epoch, which is what our test asserts.
- Pino's child bindings merge into the root `base` — no override; if a caller re-sets `service`, the root wins. Fine for our use.
- The canonical-casing test passes because Node `fetch` lowercases headers before transport; Hono's `c.req.header()` is case-insensitive either way.
