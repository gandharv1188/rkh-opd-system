## 2. How drift is modeled

The malformed payload is a ClinicalExtraction with **`document_type`
removed** but all other required fields present. This isolates schema
drift (missing required top-level field) from incidental JSON parse
errors.

The retry-once cue is asserted by inspecting the second call's
`messages` content for the stricter re-prompt language (regex:
`/json|schema|strict/i`).

## 3. Adaptation ‚Äî retry-once lives in the adapter, not the orchestrator

The Wave-2b brief hypothesised the retry-once contract might be in the
orchestrator. In the current code it is owned by the structuring
adapter itself ‚Äî see `dis/src/adapters/structuring/claude-haiku.ts`
lines 99‚Äì129. `FakeStructuringAdapter` in `tests/helpers/fake-adapters.ts`
resolves a single scripted result per call and therefore cannot model
retry on its own.

Resolution: I drive the real `ClaudeHaikuAdapter` with a scripted
`AnthropicLike` client (same pattern as the existing
`tests/unit/adapters/claude-haiku.test.ts`). This exercises the full
drift-detection path ‚Äî prompt construction ‚Üí model response ‚Üí JSON
parse ‚Üí schema validate ‚Üí retry with stricter cue ‚Üí final error ‚Äî with
no live Claude call. All file system reads for prompts + schemas are
satisfied from the bundled fixtures shipped with DIS-051.

If/when retry-once is promoted to the orchestrator layer (proposed
DIS-051-followup), the test can be simplified to use
`FakeStructuringAdapter` with a scripted list of results and assert
`.calls.length === 2`.
