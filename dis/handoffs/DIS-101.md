## 1. What was built

- `dis/src/http/middleware/error-handler.ts` — new, broader `onError` handler. Maps domain errors to the canonical envelope:
  - `HttpError` → pass through `status` / `code` / `message` / `retryable` / `details`.
  - `VersionConflictError` → 409 `VERSION_CONFLICT`, `retryable: false`, details `{ current_version, current_status }`.
  - `ExtractionNotFoundError` → 404 `EXTRACTION_NOT_FOUND`, `retryable: false`.
  - `OrchestratorError` code `IDEMPOTENCY_KEY_CONFLICT` → 409, `retryable: false`.
  - `OrchestratorError` code `UNSUPPORTED_MEDIA` → 415, `retryable: false`.
  - Any other `OrchestratorError` → 500, `retryable: true`.
  - Anything else → 500 `INTERNAL`, generic message (never leak raw `err.message`), `retryable: true`.
- Optional `deps.logger` — if provided, logs `{ err, correlation_id }` + `'unhandled-error'` tag once per request. If omitted, logging is silently skipped (useful for tests that don't inject a logger).
- `dis/tests/http/error-handler.test.ts` — 10 tests covering all mapping cases, HttpError passthrough + details echo, logger spy injection, correlation_id propagation from `x-correlation-id` header.

## 3. Decisions taken during implementation

### D-1: Keep the existing `errorEnvelope()` handler in place

**Context:** `dis/src/http/middleware/error-envelope.ts` (DIS-005) already ships an `onError` handler that knows only `HttpError`. DIS-101 is broader — it also maps orchestrator errors.
**Decision:** leave the old handler untouched for compat. The new `errorHandler()` is a superset that callers can adopt when they want domain-error mapping. `server.ts` wiring is out of scope per the brief (orchestrator will swap after merge).
**Revisit if:** we consolidate handlers; at that point `errorEnvelope()` can be deleted and any caller can migrate to `errorHandler({})` since the `HttpError` branch is identical.

### D-2: Structural logger interface, not a direct pino type import

**Context:** the brief left the logger type open (`pino` was mentioned tentatively). The handler only needs `logger.error(obj, msg)`.
**Decision:** export a minimal `ErrorHandlerLogger` interface (`{ error: (obj, msg?) => void }`). Accepts the project's real pino logger (via structural typing) and a vitest spy with equal ease. No pino dependency added to this module.
**Revisit if:** we want full typed pino context (child loggers, level, bindings) at the middleware layer — out of scope here.

### D-3: Omit `deps.logger` = silent, not default-root-logger

**Context:** the brief says "Logger is optional — if not provided, skip logging."
**Decision:** no fallback to `getRootLogger()`. Tests that don't pass a logger stay noise-free; production wiring in `server.ts` should always pass the root/request logger.
**Revisit if:** the real wiring forgets to inject one — we'd catch it via log volume, not a crash.

### D-4: `retryable` for generic `OrchestratorError` → true

**Context:** "OrchestratorError (other) → 500 internal (retryable)" per the brief.
**Decision:** yes, `retryable: true` for the fallback 500 path. Concrete known-unretryable codes (`VERSION_CONFLICT`, `UNSUPPORTED_MEDIA`, `IDEMPOTENCY_KEY_CONFLICT`) are short-circuited earlier and marked non-retryable.
**Revisit if:** we discover an orchestrator error class that encodes `retryable` on itself — we'd read it directly rather than derive from status.

### D-5: `status: number` cast to satisfy Hono's overloaded `c.json`

**Context:** Hono's `c.json` is overloaded on status-code literal types. The `OrchestratorError` branch composes a status from three literals (`409 | 415 | 500`); tsc then refused to let us compare that union against `429`.
**Decision:** widen the local `status` binding to `number` and cast to `Parameters<typeof c.json>[1]` at the call site (same pattern used by the existing `errorEnvelope()` handler). Keeps the handler API clean without introducing a status-code helper module.

## 4. What was deliberately NOT done

- **No wiring in `server.ts`.** Brief says orchestrator owns that after merge.
- **No deletion of `errorEnvelope()` (DIS-005).** Preserve for compat; future consolidation ticket can swap.
- **No enrichment of non-`OrchestratorError` thrown values** (e.g., native `ZodError`, `TypeError` from JSON parsing). They fall through to the generic `INTERNAL` branch; if the team wants per-class mappings later, they slot in before the fallback.
- **No request_id field.** The envelope shape here matches the DIS-005 body minus `request_id` (which was never populated there either). Can be added in a follow-up once we decide whether request_id and correlation_id are the same identifier.

## 5. Follow-ups / known gaps

- **Envelope consolidation (still open from DIS-029).** Three envelope shapes now coexist: `core/error-envelope.ts` (minimal, `INTERNAL_ERROR`), `http/middleware/error-envelope.ts` (`HttpError`-only, `INTERNAL`), and this new handler (broader, `INTERNAL`). A future cleanup should pick one and converge. Ticket DIS-029's §5 already flagged this.
- **ZodError mapping.** If input validation is added via zod, we'd want `ZodError → 400 VALIDATION_FAILED` with per-field details. Trivial branch to add.
- **Production wiring.** Follow-up ticket should swap `app.onError(errorEnvelope())` → `app.onError(errorHandler({ logger }))` in `server.ts` once the DIS-101 branch merges.

## 10. Relationship to existing `errorEnvelope()` (DIS-005)

Both live under `dis/src/http/middleware/`:

| Aspect              | `errorEnvelope()` (DIS-005)     | `errorHandler()` (DIS-101, this) |
| ------------------- | ------------------------------- | -------------------------------- |
| `HttpError`         | yes (native)                    | yes (delegated branch)           |
| Orchestrator errors | no — fall through to INTERNAL   | yes — mapped explicitly          |
| Logger injection    | no                              | yes (optional)                   |
| Intended callers    | early HTTP paths, pre-orch      | server.ts after orchestrator lands |

The DIS-101 handler is a strict superset of the DIS-005 behaviour for `HttpError`; anything that currently passes under DIS-005 will behave identically under DIS-101. Consolidation into a single handler is deferred.

## 11. Safety / leak review

- Generic `new Error('boom')` → response message is the literal string `Internal server error.`; original `'boom'` appears only in the injected logger call (server-side only). Verified by test case 1.
- No stack trace ever appears in the response body (no `err.stack` accessed anywhere in this module).
- Orchestrator errors DO pass their message through, but those messages are constructed in `dis/src/core/orchestrator.ts` and are caller-safe (they describe resource IDs and versions, not internal state).
