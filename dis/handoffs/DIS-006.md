## 1. What was built

- `dis/src/core/schema/clinical_extraction.v1.json` — JSON Schema draft-07. Locks the ClinicalExtraction object shape from TDD §11 verbatim: `document_type` enum, `summary`, `document_date`/`test_date`/`date_given` constrained to `^\d{4}-\d{2}-\d{2}$`, labs/medications/diagnoses/vaccinations arrays with required confidence ∈ [0,1], and `additionalProperties: false` at every level so the adapter cannot silently smuggle new fields into POC.
- `dis/src/core/schema/ajv.ts` — thin wrapper over Ajv v8. Exports `compileSchema(id)` (caches `ValidateFunction` by id), `validate(id, data)` → `{valid, errors?}`, `assertValid(id, data)` (throws), and typed `SchemaValidationError` that joins Ajv's `instancePath` + `message` lines. Schema registry is a static `SCHEMA_FILES` map so unknown ids throw at compile-time request rather than at file-read time.
- `dis/tests/unit/schema-validator.test.ts` — 10 cases: JSON load, compile caching identity, happy path, enum rejections (document_type, lab flag), missing required `confidence`, out-of-range confidence, wrong date format, minimal-empty-arrays acceptance, unknown schema id → `SchemaValidationError`.

## 3. Decisions taken during implementation

### D-1: Draft-07 (not 2020-12)

**Context:** Ajv v8 supports both; DIS-006 is indifferent.
**Decision:** Draft-07.
**Reason:** Draft-07 is the long-stable Ajv default, has no tuple/`$dynamicRef` surprises, and the schema only needs object/enum/pattern/number-range — all pre-2020. Using `ajv/dist/2020` would add a transitive export surface we do not need yet.
**Revisit if:** we ever need `prefixItems`, `dependentSchemas`, or `unevaluatedProperties`.

### D-2: `additionalProperties: false` at every level

**Context:** The Structuring adapter is non-deterministic; a prompt drift could surface unexpected keys.
**Decision:** Lock every object shape.
**Reason:** Aligns with CS-7 (verification UI must render a known set of fields) and CS-11 (promotion SQL binds specific columns). Fail-fast in POC is cheaper than silent field loss.
**Revisit if:** we add optional provider-specific diagnostics (e.g. `_debug`); then relax at the top-level only.

### D-3: `import { Ajv as AjvClass } from 'ajv'` (named, not default)

**Context:** Under NodeNext + Ajv v8 CJS package, `import Ajv from 'ajv'` typechecks as the namespace not the class, so `new Ajv()` fails TS2351.
**Decision:** Use the named `Ajv` re-export.
**Reason:** Ajv's own `.d.ts` exports both `Ajv` class and `default`; the named import is unambiguous and needs no `esModuleInterop` gymnastics.
**Revisit if:** Ajv ships an ESM build that normalises the default export.

## 4. What was deliberately NOT done

- No other schemas (only `clinical_extraction.v1`). Out of scope.
- No HTTP wiring (schema is consumed by the Structuring adapter in DIS-035/036, not here).
- No business evaluation (e.g. cross-checking that `test_category` is consistent with `test_name_normalized`). That is a promotion guard.
- No schema-evolution tooling (`v2` migration, diff). Per TDD §11: "migrations create a new version, never edit in place" — enforced by convention + CI guard when the time comes.
- No JSON Schema `$ref` to external files — inlined `definitions` to keep the file self-contained for the Edge-Function skill consumer.
