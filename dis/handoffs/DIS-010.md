## 1. What was built

- `dis/src/core/env.schema.ts` — zod v4 object schema for all DIS env vars. Numeric fields use `z.coerce.number()` so strings from `process.env` coerce in-place. `DIS_KILL_SWITCH` is a `z.string().default('false').transform(...)` returning `boolean` — accepts `"true"` and `"1"` as truthy. Cross-field rules enforced via `superRefine`: when `DIS_STACK='supabase'`, both `SUPABASE_URL` (URL-parsed) and `SUPABASE_SERVICE_ROLE_KEY` are required; when `DIS_OCR_PROVIDER='datalab'`, `DATALAB_API_KEY` is required. `ANTHROPIC_API_KEY` is always required.
- `dis/src/core/env.ts` — `loadEnv(source = process.env)`. Parses via `envSchema.parse`. On `ZodError` re-throws typed `EnvValidationError` whose message lists `  - <path>: <message>` for every issue, so an ops engineer sees all failures in one boot log line.
- `dis/tests/unit/env.test.ts` — 13 cases: defaults, numeric coercion, boolean variants (`"true"`, `"1"`), missing `ANTHROPIC_API_KEY`, stack-scoped requirements, stack=aws relaxation, ocr-scoped requirement + relaxation, bad enum values (`NODE_ENV=staging`, `DIS_STRUCTURING_PROVIDER=opus`), non-URL `SUPABASE_URL`.

## 3. Decisions taken during implementation

### D-1: `superRefine` over `z.discriminatedUnion`

**Context:** Two independent cross-field conditions: (a) stack→Supabase keys, (b) OCR provider→Datalab key. A `discriminatedUnion` supports exactly one discriminator; expressing two would require a Cartesian product (supabase+datalab, supabase+claude, aws+datalab, …) with duplicated properties across every branch.
**Decision:** Single flat `z.object` + `superRefine`.
**Reason:** Simpler schema, lower maintenance cost, error paths point directly at the offending var instead of a union mismatch.
**Revisit if:** a third conditional axis arrives (e.g. queue provider) — at that point consider a small composition helper.

### D-2: `z.coerce.number()` for numeric env vars

**Context:** `process.env.PORT` is a string; a bare `z.number()` would fail on perfectly valid input.
**Decision:** `z.coerce.number().int().positive()` (or `.nonnegative()` for minimum-chars-per-page).
**Reason:** Coercion semantics match how every Node app reads env, keeps call sites free of `Number(…)` shims.

### D-3: `ANTHROPIC_API_KEY` is unconditionally required

**Context:** TDD §14/§16 and the brief both treat Anthropic as always-on (structuring provider, visit summaries, prescription generation).
**Decision:** `z.string().min(1)` with no conditional wrapper.
**Reason:** Booting without an Anthropic key guarantees every pipeline run fails — fail at boot, not at first request.
**Revisit if:** a non-Claude structuring option (`DIS_STRUCTURING_PROVIDER='onprem'`?) lands.

### D-4: `DIS_KILL_SWITCH` accepts `"true"` and `"1"`, defaults to `"false"`

**Context:** Ops engineers type either in shells; we should not have a footgun where `DIS_KILL_SWITCH=1` silently stays off.
**Decision:** Treat `"true"` OR `"1"` as true; everything else (including empty) as false.
**Revisit if:** we need three-state (on/off/shadow) — swap to an enum.

## 4. What was deliberately NOT done

- No secrets retrieval (DIS-055, SecretsPort). This loader reads plain `process.env` only.
- No runtime reloading / hot config. Boot-time validation only per §16.
- No `.env` file loading (`dotenv` etc.) — that is the shell's or the deploy harness' job. Node 20's `--env-file` flag is the recommended local path.
- No logging of the parsed env. Doing so would leak `SUPABASE_SERVICE_ROLE_KEY`/`ANTHROPIC_API_KEY` into logs; §16 forbids it.
