## 1. What ships

An end-to-end Vitest case that wires `createServer()` with both the
kill-switch middleware and the real `/ingest` route, then flips the
switch mid-session and asserts the next `POST /ingest` is short-circuited
with `503 + Retry-After + KILL_SWITCH_ACTIVE` within a single request —
confirming there is no cached "switch was off when we started" state.

File:

- `dis/tests/e2e/kill-switch.test.ts` — one test, four phases.

## 2. Why this ticket exists

DIS-100 asserts middleware behaviour in isolation against a stub Hono
app. DIS-108 closes the integration gap: it proves the switch takes
effect **through the real `createServer()` composition root**, on the
actual `/ingest` route, and **mid-session** — the scenario ops would
actually trigger during an incident (an operator toggles a row in
`dis_kill_switch`; subsequent writes must fail immediately without a
process restart).

## 3. Test shape

Four phases in one `it('subsequent POST returns 503', …)`:

1. `killSwitchOn = false` → `POST /ingest` → `201` from the fake
   orchestrator.
2. Mutate the closure: `killSwitchOn = true`. No server restart.
3. `POST /ingest` with a fresh `Idempotency-Key` → `503`,
   `Retry-After` header present, envelope `error.code ===
   'KILL_SWITCH_ACTIVE'`.
4. `GET /health` → `200` (reads bypass kill-switch, per DIS-100).

The closure over `killSwitchOn` is deliberate: the middleware receives
`isEnabled: () => killSwitchOn` and re-reads on every request, so the
test is the live wiring contract.

Final assertion: `fakeOrchestrator.calls.length === 1` — proves the
short-circuit happened **before** the route handler ran, not after.

## 4. Dependency wiring

- `createServer({ killSwitch: { isEnabled }, routes: { ingest: { orchestrator } } })`.
- `FakeOrchestrator` implements only `ingest(input)` → returns a minimal
  `ExtractionRecord`, cast through `unknown` to `IngestionOrchestrator`
  (same pattern used in `tests/http/ingest.test.ts`).
- No DB, no ports, no real adapters — this is a flip-switch assertion,
  not a state-machine test.

## 5. What CS-9 asks for (and what this test answers)

CS-9 requires a tested, operator-controllable emergency stop for
ingestion writes that (a) takes effect immediately, (b) leaves reads
alive, (c) returns a retryable envelope. DIS-108 supplies the
immediate-takes-effect leg. Sign-off is batched pre-Epic-G with
DIS-100/101/102.

## 8. Risks / non-goals

- **Non-goal:** DB-backed `isEnabled` (covered by DIS-100 §"uses the
  DatabasePort…").
- **Non-goal:** rate-limit or error-handler interaction (separate Epic-D
  tickets).
- **Risk:** the closure-mutation pattern relies on `isEnabled` being
  called per request. If anyone adds caching to `killSwitch()`, this
  test becomes the canary — it will fail. Intended.
