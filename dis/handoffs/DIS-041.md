## 1. What was built

`dis/tests/integration/content-hash-dedupe.test.ts` composes `sha256()`
from DIS-027 with the `IngestionOrchestrator.ingest()` idempotency path.
The caller wires `sha256(bytes)` as the Idempotency-Key; byte-identical
submissions therefore collide on that key and orchestrator returns the
existing extraction with zero additional side effects on storage.

Cases:

1. `sha256` is deterministic over identical bytes; digest matches
   `/^[0-9a-f]{64}$/`.
2. Different bytes produce different digests (sanity).
3. Two `ingest()` calls with byte-identical payload + the same
   content-hash key → same extraction id + version; `storage.puts.length
   === 1`; `db.insertExtraction` called exactly once.
4. Reusing the hash of payload A as the key for payload B (caller
   lies about content hash) surfaces `IDEMPOTENCY_KEY_CONFLICT` and no
   second `putObject` occurs.

## 3. Decisions

### D-1: Use `FakeStorageAdapter.puts` from helpers, not `FakeStorage` from `__fakes__`

The `__fakes__` variant exposes `.objects` (a Map) but not a call-count
array. The helpers variant tracks `.puts: PutObjectInput[]`, which is
what the ticket's VERIFY language ("only ONE putObject call") needs.

### D-2: Caller wires content hash → Idempotency-Key (not orchestrator)

The orchestrator's `hashPayload()` is a deterministic non-crypto digest
(filename+size+content-type+patient) that lives inside `payload_hash`,
not a crypto hash of the body. DIS-027 owns the crypto hash; the
integration contract this ticket verifies is "caller passes
sha256(bytes) as the idempotency key", which is the real dedupe
mechanism.

### D-3: Collision case distinct from dedupe

Case 4 tamper scenario (same key, different bytes) asserts the
orchestrator's `IDEMPOTENCY_KEY_CONFLICT` guard — the safety net for a
client that lies about the content hash. Dedupe (case 3) and collision
(case 4) are two different branches of the same code path.
