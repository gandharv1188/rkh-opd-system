## 1. What was built

`dis/tests/integration/correlation-propagation.test.ts` binds a fixed
correlation id via `withCorrelation('abc-123', async () => { ... })`
and asserts it propagates through the full observability surface of a
single ingest:

- (a) `currentCorrelationId()` returns `'abc-123'` both before and
  after `await orch.ingest(...)`, proving AsyncLocalStorage survives
  the async continuation.
- (b) `AuditLogger.write(event)` inserts with the correlation id as
  the last INSERT parameter (position matches `audit-log.ts`
  `INSERT_SQL` — 11th/last column).
- (c) `metrics.inc('dis_ingest_total', { correlation_id, outcome })`
  records a labelled sample (asserted via `snapshot()`), AND
  `logger.info({ correlation_id }, 'ingest ok')` emits a structured
  JSON line containing `"correlation_id":"abc-123"` (captured via a
  `Writable` sink injected into `createLogger`).

A second `it()` verifies nested `withCorrelation` scopes override and
restore correctly.

## 3. Decisions

### D-1: Compose utilities directly inside the `withCorrelation` scope

The orchestrator itself does not (yet) emit audit/metrics/log lines
per-ingest — those sinks are wired at the HTTP + router layer. For the
purpose of verifying the DIS-028 utility surface, the test composes
`AuditLogger`, `createMetrics`, and `createLogger` directly inside the
scope. This is the contract that callers of `withCorrelation` rely on
("inside my callback, `currentCorrelationId()` returns the bound id").
A future ticket can add an orchestrator-level audit-on-ingest hook and
assert propagation end-to-end without re-writing this test.

### D-2: Assert audit `correlation_id` position by INSERT param index

`audit-log.ts` INSERT_SQL binds 11 parameters, with `correlation_id`
last. Asserting `auditParams[auditParams.length - 1] === id` is
robust to re-ordering of earlier params but breaks loudly if
`correlation_id` is dropped from the INSERT — exactly the regression
this test guards against.

### D-3: Substring-match on logger JSON instead of re-parsing

pino emits `{"level":30,"time":...,"correlation_id":"abc-123",...}`.
Substring matching on `"correlation_id":"abc-123"` is sufficient and
avoids coupling to pino's line layout (timestamps, pid, level codes).
