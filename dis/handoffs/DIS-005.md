## 3. Decisions taken during implementation

### D-1: `HttpError` class over a tagged-union record

**Context:** handlers need a cheap way to signal HTTP-shaped failures.
**Options considered:** thrown `HttpError` subclass, returned `Result<Envelope>`, tagged-union objects.
**Decision:** `HttpError extends Error` thrown into `onError`.
**Reason:** matches Hono's built-in `onError` dispatch; lets any deep-called helper short-circuit without plumbing result types through every layer.
**Revisit if:** we move to a Result-style core that forbids exceptions — unlikely for HTTP glue.

### D-2: Default `retryable` by status

**Context:** error_model.md declares 5xx, 429, 502, 504 as retryable.
**Decision:** `defaultRetryable(status)` maps 429 + any 5xx → `true`; caller can override.
**Revisit if:** 501 ever needs non-retryable semantics.

### D-3: Unknown-error message suppression

**Context:** §8 security — never leak raw server messages.
**Decision:** unknown `Error` → generic `"Internal server error."` body with the raw message only in logs (DIS-008 wires pino later).
**Revisit if:** we add a dev-only verbose mode; gate behind env.

### D-4: Router convention is a thin helper, not a DSL

**Context:** ticket scope is "routing convention", not a framework.
**Decision:** `RouteModule` type + `registerRoutes()` iterating an ordered list. Feature files keep their current `registerXRoute(app)` shape.
**Revisit if:** we need nested sub-apps or per-feature middleware chains.

## 10. Non-obvious gotchas

- Hono's `onError` handler returns `Response`; `c.json(body, status)` requires the status to be a valid Hono status literal. Cast via `Parameters<typeof c.json>[1]` keeps type safety when `status` is a runtime number from `HttpError`.
- `correlationId` middleware MUST be registered before `onError` binds, since the handler reads `c.get('correlationId')`. Already the case in `createServer()`.
