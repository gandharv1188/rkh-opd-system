# DIS-106 — E2E idempotency replay

## 1. Summary

Added an end-to-end test that fires the same `Idempotency-Key` (with identical
body) five times through the full HTTP stack and asserts exactly one
extraction row is created and one storage put is performed. Proves DIS-090
(ingest route) + DIS-025 (idempotency store) compose correctly.

## 2. Scope

- New file: `dis/tests/e2e/idempotency.test.ts`
- No production code changes.
- No fake-adapter changes — `FakeDatabaseAdapter.findExtractionByIdempotencyKey`
  already walks `.rows`, so an in-memory store is not needed.

## 3. Design decisions

- Wired a real `IngestionOrchestrator` (not a fake) to exercise the actual
  dedupe branch in `orchestrator.ingest()`. The fake orchestrator used by
  `tests/http/ingest.test.ts` would have been a weaker contract (it cannot
  miss a dedupe bug in the orchestrator itself).
- Used `createServer({ routes: { ingest: { orchestrator } } })` to get the
  full middleware stack (correlation-id, error-handler) around the route —
  matches how production boots.
- Body chosen so `hashPayload()` is stable across replays (it already is —
  hash is `patientId:filename:contentType:length`).

## 4. Test spec (actual behaviour)

```
POST /ingest x 5  (same headers, same body, same Idempotency-Key)
  → all 5 return 201
  → response extraction_id is identical across all 5 (set size = 1)
  → db.rows.length === 1
  → storage.puts.length === 1
```
