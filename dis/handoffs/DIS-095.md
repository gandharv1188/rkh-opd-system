# DIS-095 — GET /extractions (queue listing)

## 1. Summary
Cursor-paginated list route at `GET /extractions` filtered by `status` / `patient_id`.
Returns `{ items, next_cursor, correlation_id }`. Limit defaults to 50, hard-capped
at 200. `next_cursor` is the id of the last row when more rows exist, else `null`.

## 4. Route contract
- Query: `status`, `patient_id`, `cursor`, `limit`.
- Cursor = last `id` from the previous page; clause is `id > $N ORDER BY id ASC`.
- Fetches `limit+1` rows to detect "more exist"; trims to `limit` in response.
- `limit <= 0` or non-numeric → `HttpError(400, 'INVALID_LIMIT')`.
- `limit > 200` is clamped silently to 200 (hard cap).

## 5. SQL literal location
The SQL literal (`SELECT * FROM ocr_extractions …`) lives in
`dis/src/http/routes/extractions-list.ts`. Fitness rule `core_no_sql_literals`
only scans `dis/src/core/**`, so this route is outside its glob — no violation.
Per the DatabasePort contract (§Database port / DRIFT-PHASE-1 §5 FOLLOWUP-A),
parameterised SQL at the adapter / HTTP boundary is the accepted pattern;
see `dis/src/ports/database.ts` lines 54-63.

## 6. Schema limitation — `operator_id` filter omitted
The backlog's §3 spec calls for an `operator_id` filter, but the base migration
`dis/migrations/M001_ocr_extractions.sql` does not define an `operator_id`
column. Closest existing fields are `uploader_id` and `verified_by` — neither
has the exact semantics of "operator who owns this queue entry". Per the
brief's degrade-not-extend rule, the filter is simply omitted. When a future
migration adds `operator_id` (or the product decides on `uploader_id` as a
synonym), add the filter clause and a matching test — the parameter-binding
pattern is already established.

## 7. Test doubles
`makeDb()` in the test file provides a `DatabasePort` whose `query()` records
the `(sql, params)` pair and returns caller-scripted rows. Other port methods
throw on use so accidental exercise fails loudly. `insertExtraction` has a
typed wrapper rather than `notUsed` directly to satisfy the `InsertExtractionInput`
signature.

## 8. RLS / security posture
The route does not call `setSessionVars` directly — consistent with sibling
routes (`extractions-get`, `extractions-approve`, etc.), the adapter layer
applies `app.role` / `app.patient_id` before issuing queries. Cross-scope rows
will not appear in results. No existence leak: the listing cannot distinguish
"no rows match" from "all rows blocked by RLS".

## 9. Correlation id
`correlation_id` is read from the Hono context (`c.get('correlationId')`) and
echoed in the response body, matching the envelope/ingest pattern.
