# DIS-072 — Shared StructuringPort contract test suite

**Status:** Done
**Branch:** `feat/dev-c-wiring-contracts`
**Wave:** 3a (Epic C)
**TDD ref:** §10.1, §11
**Files:** `dis/tests/integration/structuring-port-contract.test.ts`

## 1. What

Parameterized contract suite `runStructuringPortContractSuite(factory)` that
every `StructuringPort` implementation re-runs. Pins the port-level invariants
so new providers (sonnet / opus / onprem) inherit correctness checks.

Applied against two implementations:

- `FakeStructuring` (DIS-012) — canned StructuringResult; `retryOnce` returns
  `null` because the fake does not simulate retries (suite skips cleanly).
- `ClaudeHaikuAdapter` — scripted `AnthropicLike` factory (no SDK / network);
  retry test observes two provider calls.

## 2. Contract assertions

Per `TDD §10.1`, `§11`, `adapters.md §Ground rules`, `DIS-051`:

1. `StructuringResult` shape — provider, providerVersion, tokensUsed,
   costMicroINR (≥ 0), latencyMs (≥ 0), structured, rawResponse.
2. `structured` passes ClinicalExtraction v1 required-keys check —
   `document_type`, `summary`, `labs`, `medications`, `diagnoses`,
   `vaccinations`. (Full JSON-Schema validation lives in the adapter's own
   test; the contract pins the minimum bar at the port boundary.)
3. `rawResponse` is present and non-null (CS-2 spirit — structuring replies
   are stored for audit; the exact byte-equality guarantee is weaker here
   than for OCR because the SDK wraps the provider's JSON).
4. `tokensUsed`, `costMicroINR`, `latencyMs` are non-negative finite numbers.
5. Retry-once-on-schema-invalid: when the first reply is not schema-valid,
   the adapter retries once and the second reply is returned. Observable as
   `retryOnceCallCount() === 2`.

## 3. Decisions

- **File location.** Brief directs
  `dis/tests/integration/structuring-port-contract.test.ts`; backlog called
  for a `.contract.ts` + `.runner.test.ts` split — same divergence as
  DIS-071, same rationale (vitest include pattern + single-file
  deliverable). Follow-up DIS-072-followup-b can refactor.
- **Schema assertion strength.** Used a required-keys check, not full
  Ajv validation against `clinical-extraction.v1.json`. Reasons: (a) the
  adapter's own unit test (`tests/unit/adapters/claude-haiku.test.ts`)
  already validates specific field shapes; (b) keeping the contract lean
  lets fakes return minimal-valid payloads without dragging Ajv into the
  suite. A stricter contract variant can be added under
  DIS-072-followup-a once DIS-030's `validateExtraction` is stable.
- **Retry-case opt-out.** `retryOnce(): StructuringPort | null` — fakes
  return `null`, suite early-returns. Avoids forcing fakes to simulate
  provider-level retry behavior that only real adapters implement.
- **Provider-call counter.** Optional factory hook `retryOnceCallCount`.
  When present, asserts `=== 2`. Keeps the contract strict on
  retry-count without coupling to adapter internals.
- **`structured` nullability.** `ClaudeHaikuAdapter.structure` can set
  `structured: null` if both attempts fail (see src line 139) — but only
  on the way to throwing `StructuringSchemaInvalidError`. The happy-path
  test never exercises that branch, so `expect(structured).not.toBeNull()`
  is safe.
