## 3. Decisions taken

### D-1: Count `labInserts.length` rather than fake-DB query calls

**Context:** Brief says "Use FakeDatabaseAdapter (DIS-012) to count inserts." But `buildPromotionPlan` is a pure intent-builder that emits a `PromotionPlan` — it doesn't call the DB at all. The only "insert count" available from the plan is `plan.labInserts.length`.
**Decision:** Assert against `plan.labInserts.length`. Use `FakeDatabaseAdapter.calls` as a secondary invariant — it must stay at 0, proving the plan-builder is pure.
**Reason:** Matches the actual architecture (§13 TDD separates plan-building from plan-execution). When DIS-054 lands the executor, a companion integration test can count real `ocr_audit_log` / `lab_results` insert SQL.

### D-2: Use the same 7-reading fixture to exercise both CS rules and a negative control

**Reason:** Isolates behaviour differences to a single variable at a time (existing hints → CS-11; document_type → CS-10 scoping). Easier to reason about when the test fails in the future.

### D-3: Exact pkid reconstruction in-test

**Context:** `promotion.ts` builds `labKey` as `${patientId}|${testName}|${testDate}|${valueNumeric}` — an internal detail.
**Decision:** Replicate the format in a local `pkid()` helper rather than importing the unexported `labKey`. Documented via comment on the helper.
**Reason:** Keeps the test as a true "black-box integration" — if the internal key format ever changes, the test fails loudly, which is the right signal (promotion's pkid *is* the CS-11 contract).
