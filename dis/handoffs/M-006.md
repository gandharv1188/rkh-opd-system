## 1. What ships

Additive columns on the two clinical tables DIS promotes into:

- `ocr_extraction_id uuid REFERENCES ocr_extractions(id) ON DELETE RESTRICT`
  (nullable — legacy rows stay null forever)
- `verification_status text NOT NULL DEFAULT 'legacy'` with CHECK in
  `('legacy','verified','ai_extracted','auto_approved','manual')`
- `verified_by text`, `verified_at timestamptz`

Backfill: every pre-existing row gets `verification_status='legacy'` via
two idempotent `UPDATE ... WHERE verification_status IS NULL` statements.

## 2. CRITICAL: live-application gate

`lab_results` and `vaccinations` are production POC tables. This migration
is **additive only** — but the brief explicitly calls out that it touches
existing tables. The rules:

1. Apply to `dis_test` (staging) in Wave 7 only.
2. Do NOT apply to the live Radhakishan Supabase instance until the
   clinical-safety gate in rollout_plan.md Stage C is passed (sign-off from
   Dr. Lokender Goyal + user confirmation).
3. If a follow-up ticket ever needs to make `ocr_extraction_id` NOT NULL,
   that is M-009 (cutover migration), which requires a full rollout review.

## 3. Decisions

- **Default `'legacy'`, not `'verified'`.** data_model.md's example used
  `'verified'` as the default; the brief said `'legacy'`. Chose the brief's
  value because "legacy" is unambiguous for rows pre-dating DIS —
  `'verified'` risks overclaiming nurse sign-off that never happened.
- **CHECK constraint includes `'legacy'`** as a valid state, so the
  pre-DIS data is not in constant constraint violation.
- **Backfill scope:** only rows where `verification_status IS NULL` (ie
  never seen before). After the first run the UPDATE matches nothing —
  idempotent.

## 4. Test plan

- Static parse — asserts ALTER + ADD COLUMN + default `'legacy'` + UPDATE.
- Live (Wave 7, staging only): snapshot existing row count → apply → assert
  count unchanged + every row has non-null verification_status → rollback
  → assert columns removed → re-apply.

## 6. Gotchas

- `ADD COLUMN IF NOT EXISTS` + `NOT NULL DEFAULT` — on older PG versions
  this would rewrite the table. PG 11+ short-circuits when the default is
  non-volatile, so on PG 16 this is a metadata-only change. Verify on
  the target Postgres version before running in prod.
- The POC `vaccinations` table has no `source` column, so its backfill
  cannot mirror `lab_results`' `source='manual' → 'manual'` mapping from
  data_model.md. We collapse everything pre-DIS to `'legacy'` uniformly;
  a later migration can refine if needed.
