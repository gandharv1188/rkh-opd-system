# DIS-148 — Metrics gauges + counters

## Scope
Wired the 5 canonical named metrics (TDD §14) on top of the existing in-memory `Metrics` port at `dis/src/core/metrics.ts`.

## Files
- `dis/src/observability/metrics.ts` (new) — typed facade: `METRICS`, `MetricName`, `setGauge`, `incCounter`, `observeLatency`, `snapshotNamedMetrics`.
- `dis/tests/observability/metrics.test.ts` (new) — 3 tests covering presence of all 5 series, default-to-zero behaviour, and gauge-replace vs counter-accumulate semantics.

## API surface exposed

- `METRICS.EXTRACTIONS_APPROVED_TOTAL` → `extractions_approved_total` (counter)
- `METRICS.EXTRACTIONS_REJECTED_TOTAL` → `extractions_rejected_total` (counter)
- `METRICS.OCR_LATENCY_MS_P95` → `ocr_latency_ms_p95` (latency observation; last-value)
- `METRICS.COST_MICRO_INR_TOTAL` → `cost_micro_inr_total` (counter)

## Adaptation note (important for reviewers)
The real `Metrics` port only exposes `inc(name, labels?, by?)` and `snapshot()` — not `setGauge` / `incCounter` / `observeLatency` as the brief's example code assumed. Per brief guidance ("if method names aren't the exact ones, adjust to what's there"), the facade maps:
- Counters → `m.inc(name, undefined, delta)`; `snapshotNamedMetrics` sums counters by name from `m.snapshot().counters`.
- Gauges → side-table keyed by `Metrics` instance (WeakMap) so `setGauge` replaces prior value. The core port's `snapshot().gauges` is always `[]` today.
- Latency → same side-table strategy; stores last observation. P95 aggregation is expected upstream — the caller passes the already-computed P95 value.

When DIS-009 is extended to support native gauges/histograms, the side-tables in this file can be deleted and the facade flipped to call through directly. Tests don't depend on the storage strategy.
