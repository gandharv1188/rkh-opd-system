## 3. Decisions taken during implementation

### D-1: Minimal envelope shape, not the full HTTP body

**Context:** `dis/src/http/middleware/error-envelope.ts` already defines `ErrorEnvelopeBody` with `retryable` + `request_id`. The brief for DIS-029 specifies the narrower shape `{ code, message, correlation_id, details? }`.
**Decision:** keep the core builder minimal as the brief dictates. HTTP middleware continues to own its own wrapper and can layer `retryable` / `request_id` on top when it composes the core output (or continue to build the HTTP body independently â€” this builder is for non-HTTP callsites like queue workers, audit sinks, and log enrichers).
**Revisit if:** we consolidate on a single envelope shape across HTTP + non-HTTP paths. That is a separate consolidation ticket.

### D-2: `INTERNAL_ERROR` code, per brief, not `INTERNAL` from error_model.md

**Context:** the brief explicitly specifies `INTERNAL_ERROR`; `04_api/error_model.md` uses `INTERNAL`. These differ.
**Decision:** follow the brief. VERIFY-4 does not care about this code, and the HTTP middleware has its own `INTERNAL` emission. Any caller wanting HTTP conformance should use the HTTP envelope.
**Revisit if:** a consolidation pass wants a single canonical string â€” trivial one-line swap in `EXPLICIT_OVERRIDES` + unknown-fallback.

### D-3: Name-based mapping, not `instanceof` chains

**Context:** the brief lists a derivation rule "for any class whose name ends in `Error`, strip suffix â†’ UPPER_SNAKE". That is naturally a name check.
**Decision:** read `err.name` (falls back to `err.constructor?.name`). No `instanceof` imports from adapter modules, so the builder does not take on a dependency on every adapter and adapters can evolve without ripple.
**Revisit if:** we ever want type-level exhaustiveness. A future ticket could add a small registry of known error classes with compile-time assertions.

### D-4: `"unknown"` sentinel instead of empty string

**Context:** the HTTP middleware uses `''` when no correlation ID is present. The brief for DIS-029 specifies `"unknown"`.
**Decision:** follow the brief. `"unknown"` is filterable in log aggregations and distinguishes "genuinely missing" from "explicitly empty".
**Revisit if:** the two envelopes are merged.

### D-5: `details` field omitted by default

**Context:** the `ErrorEnvelope.error.details` field is optional. Adding details requires inspecting class-specific fields (e.g., `VersionConflictError.currentVersion`), which expands scope.
**Decision:** leave `details` absent; consumers can layer it on if needed. VERIFY-3 passes without it.
**Revisit if:** a follow-up wants structured details for specific classes (would add a tiny per-class extractor map).
