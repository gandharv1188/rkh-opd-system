## 3. Decisions taken during implementation

### D-1: Mocked `mammoth` via `vi.mock` instead of committing a real .docx fixture

**Context:** The adapter test needs a DOCX byte stream. Building a minimal-valid OOXML zip in-repo (Content_Types.xml + _rels/.rels + word/document.xml, deflated + central-directory-assembled with `node:zlib`) is ~60–100 lines of fragile hand-rolled zip writer code. No `docx` npm package is installed and the brief forbids touching `package.json`.
**Options considered:** (a) hand-rolled zip to generate fixture in `beforeAll`; (b) commit a pre-built .docx Buffer literal; (c) `vi.mock('mammoth')` and verify transformation logic only.
**Decision:** (c) mock mammoth.
**Reason:** The adapter under test is a thin transformation layer — its job is mapping `mammoth.extractRawText` output onto `ExtractionResult`. That contract is fully exercised by a mock. The brief explicitly permits this: "This is acceptable." Avoids baking an undocumented binary fixture into the repo and keeps the test deterministic/offline.
**Revisit if:** integration coverage is later required across the real mammoth parse path — at that point, promote to an integration test with a real fixture (see §5 follow-up).

### D-2: `pageCount = 1` fixed constant

**Context:** `mammoth.extractRawText` returns a flat text stream; Word documents have no hard page concept in XML (pagination is a rendering concern). ADR-008 allows adapters to emit reasonable conventions here.
**Decision:** Hard-code `1`.
**Reason:** Matches the brief's documented convention; computing true page count would require rendering via a headless Word engine — far outside adapter scope.
**Revisit if:** downstream (e.g. chunking) requires genuine page counts for DOCX.

## 4. What was deliberately NOT done

- No real .docx fixture committed (see D-1). No `office/*.docx` was created.
- No integration test against mammoth's real parse path.
- No support for `.doc` (legacy binary format) — mammoth does not parse it; router should not send `.doc` here.
- No error-path tests (corrupt zip, password-protected docx). Error propagation is implicit — `mammoth.extractRawText` will reject, and our `async` adapter surface propagates the rejection unchanged.

## 10. Non-obvious gotchas

- The test imports the adapter AFTER `vi.mock('mammoth', ...)` at module top level — vitest hoists `vi.mock` so this works, but re-ordering to a conventional "imports first, mocks after" style will break it.
- `mammoth` default export is hoisted via `{ default: { extractRawText } }` in the mock — it is an ESM default-export module in its type surface. The adapter uses `import mammoth from 'mammoth'` which matches.
- `ExtractionInput.bytes` is `Uint8Array`; adapter wraps with `Buffer.from(input.bytes)` because mammoth requires a Node `Buffer` (its `{ buffer }` input is Buffer-typed).
- `pageCount = 1` is a route convention, not a property of the source document — do not try to derive it from mammoth output.
