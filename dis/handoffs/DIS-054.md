## Contract summary

- `query<T>()` → readonly (frozen) array; delegates to the driver's
  parameterised `unsafe(sql, params)` so the port's `(sql, params)` shape is
  preserved. No string interpolation of user input.
- `queryOne<T>()` → first row or `null`.
- `transaction()` → uses driver's `sql.begin(fn)` which emits
  `BEGIN`/`COMMIT` on success and `ROLLBACK` on thrown errors. The nested
  handle passed to `work` is a fresh `SupabasePostgresAdapter` bound to the
  transactional `sql`, so session vars stay transaction-scoped.
- `setSessionVars()` → emits `SET LOCAL <key> = $1` per key, parameterised.
  Key names are validated against `^[a-z_][a-z0-9_.]*$` to refuse injection
  via the identifier slot.
- Pool reuse: one `SqlClient` is held by the adapter and reused across
  calls. The default client is created lazily via `driverLoader`.

## Typed errors

- `DatabaseError` — base (`code: string`).
- `DatabaseConnectionError` — thrown for `ECONNREFUSED / ECONNRESET /
ETIMEDOUT / ENOTFOUND / EAI_AGAIN / CONNECTION_*` driver codes so callers
  can branch without importing the driver.

## Driver wiring (for integrator)

The adapter does **not** import `postgres` at module load so unit tests stay
hermetic. The wiring layer (DIS-060 / `src/wiring/supabase.ts`) must call:

```ts
import postgres from 'postgres';
import { setPostgresDriverLoader } from '../adapters/database/supabase-postgres.js';

setPostgresDriverLoader(
  (cs) => postgres(cs, { prepare: true, max: 10, idle_timeout: 30 }) as unknown as SqlClient,
);
```

`connectionString` is resolved via `SecretsPort.get('DATABASE_URL')`.

## Portability check

- No `@supabase/supabase-js` import anywhere in `dis/src/` (V4 passes).
- Queries are generic SQL; no Supabase-only extensions used.
- Same adapter file is the one production AWS/RDS will use — only the
  wiring layer differs.

## Test plan

Seven tests (≥6 required):

1. `query()` returns a frozen array, parameters flow through, no interpolation.
2. `queryOne()` returns first row.
3. `queryOne()` returns `null` on empty.
4. `transaction()` commits on success.
5. `transaction()` rolls back and rethrows on error.
6. `setSessionVars()` emits parameterised `SET LOCAL`.
7. Pool reused across multiple calls.
8. Connection errors wrapped in `DatabaseConnectionError`.
