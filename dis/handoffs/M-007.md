**Files:**

## 1. What ships

Partial unique indexes that block DIS from inserting the same
patient/test/date/value twice on lab_results, and the same
patient/vaccine/date/dose twice on vaccinations:

```
uniq_lab_dedupe ON lab_results (patient_id, test_name, test_date,
                                coalesce(value_numeric::text, value))
                  WHERE ocr_extraction_id IS NOT NULL
uniq_vax_dedupe ON vaccinations (patient_id, vaccine_name, date_given,
                                 coalesce(dose_number, 0))
                  WHERE ocr_extraction_id IS NOT NULL
```

Legacy / manually-entered rows are excluded by the WHERE clause ‚Äî a POC
user can still enter duplicate manual rows if they want; only DIS writes
are deduped.

Pre-flight dry-run inside a `DO $$ ... $$` block: counts duplicates within
the DIS partition, aborts with `RAISE EXCEPTION` listing up to 10 sample
keys if any exist. The migration does not create the index until the
check passes, so a failing environment is left untouched.

## 2. Decisions

- **Partial index, not a full one.** CS-11 asks for dedupe on DIS-inserted
  rows; a full unique constraint would immediately conflict with existing
  legacy data (which can legitimately have "duplicate-looking" rows ‚Äî
  same test retaken same day for instance).
- **Columns diverge from the brief.** Brief referenced
  `test_name_normalized`, `value_numeric`, `vaccine_name_normalized` ‚Äî none
  of those columns exist on the current POC schema. Documented in the SQL
  comment; a future migration can add normalised columns and swap the
  index. Today's columns (`test_name`, `test_date`, value coalesce;
  `vaccine_name`, `date_given`, `dose_number`) are what the POC has.
- **`coalesce(dose_number, 0)`** ‚Äî NULL dose_numbers would otherwise slip
  past a unique index (two NULLs are not equal). data_model.md uses the
  same trick.

## 5. Gotchas

- `coalesce(value_numeric::text, value)` ‚Äî if both are NULL, the coalesce
  returns NULL and the row escapes the unique index. Acceptable: a lab
  result with no value at all is a data-quality issue worth not blocking.
- The dry-run does `group by ... having count(*) > 1 limit 10` ‚Äî `limit`
  inside a subquery means the error only lists 10 sample keys, not all.
  Operators needing the full list can drop the limit and re-run the
  SELECT by hand.
