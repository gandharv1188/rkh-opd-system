## 1. What ships

Single-row-at-a-time config table. `enabled=false` by default — the pipeline
never auto-approves extractions until an admin explicitly flips it via a
newly inserted row. A partial unique index
(`uniq_dis_confidence_policy_active`) enforces at-most-one
`deactivated_at IS NULL AND enabled = true` row cluster-wide.

Seeded with `(version=1, policy='{}'::jsonb, rules='[]'::jsonb,
enabled=false)` so the runtime lookup has a row to find on day one.

## 2. Decisions

- **Superset columns again.** Brief used `policy jsonb`, data_model.md used
  `rules jsonb`. Both are nullable/defaulted so either write pattern works.
- **`activated_by uuid`** per brief (not `text` as data_model.md has).
  The app's user identifier is a UUID from the admin console; text would
  force coercion on every write.
- **Partial unique index on `(true)`** is the standard PG trick to enforce
  "at most one row matching predicate" without adding an artificial column.

## 3. Test plan

## 4. Dependencies

- None (no FKs). Can be applied in any order relative to M-001/M-002.

## 5. Gotchas

- `uniq_dis_confidence_policy_active` enforces the "one active" rule only
  when `deactivated_at IS NULL`. Switching policies must be a two-step
  transaction: `UPDATE old SET deactivated_at = now(); INSERT new row`.
- The audit trail for policy changes lives in `ocr_audit_log` (actor type
  'user', event 'policy_change') — not in this table. This table is truth
  of current state; history is in the log.
