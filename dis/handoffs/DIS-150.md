# DIS-150 — Alert Webhook (Queue Depth)

## Summary
`QueueDepthAlerter` fires a POST webhook when `queue_depth > threshold` is sustained for `sustainedMs`, deduped by `cooldownMs`.

## Files
- `dis/src/observability/alerts.ts` — `AlertPolicy`, `QueueDepthAlerter.observe()` returning `'fired' | 'noop' | 'armed'`.
- `dis/tests/observability/alerts.test.ts` — 3 tests: sustained-breach fire, cooldown dedup, dip-below-threshold reset.

## Behaviour
- `queueDepth <= threshold` → reset breach window, return `noop`.
- Fire dedup: if `lastFiredAt > 0` and `now - lastFiredAt < cooldownMs` → return `noop` (cooldown check before sustained-window check so post-fire ticks stay quiet).
- First breach tick arms (`breachStartAt = now`) → `armed`.
- Subsequent ticks within `sustainedMs` → `armed`.
- Sustained ≥ `sustainedMs` and past cooldown → POST `{ alert, threshold, current, sustained_ms, fired_at }` to `webhookUrl`, update `lastFiredAt`, reset `breachStartAt = now`, return `fired`.

## Deviations from brief
Reordered cooldown check to run before the sustained-window check. The brief's original ordering returned `armed` (not `noop`) during the cooldown window immediately after a fire, which contradicted the `"does not re-alert within cooldown"` test. Fix: early-return `noop` whenever we're inside the cooldown window, regardless of breach-window state.

## Integration notes
`policy.now` and `policy.fetchFn` are injected for tests. Production wiring: omit them to use `Date.now` and global `fetch`. Caller must invoke `observe(depth)` on every metrics tick (e.g. inside existing metrics loop); alerter is stateful, keep a singleton per webhook target.
