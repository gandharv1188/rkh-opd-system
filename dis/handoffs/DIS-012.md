## 1. What was built

Script-driven fake adapters for all 8 DIS ports, living at
`dis/tests/helpers/` (distinct from the orchestrator fakes under
`dis/src/core/__fakes__/`, which are single-behaviour). Each fake takes a
`Record<key, {success}|{error}>` "script" object and records every call on a
`.calls` / `.enqueued` / `.puts` array for assertion.

- `FakeOcrAdapter` — keyed by first-page UTF-8 bytes (filename-as-Buffer is
  the usual test shape); otherwise falls back to a deterministic
  `pages:N/mediaType` key.
- `FakeStructuringAdapter` — keyed by `documentCategory`.
- `FakeStorageAdapter` — real in-memory Map semantics for put/get/delete +
  scripted get errors; signed URL helpers return discriminated fakes.
- `FakeDatabaseAdapter` — full `DatabasePort` surface incl. the DIS-021b/d
  named methods (findExtractionById, findExtractionByIdempotencyKey,
  insertExtraction, updateExtractionStatus) with optimistic-lock semantics on
  `version`, plus `query` / `queryOne` / `transaction` / `setSessionVars`.
- `FakeQueueAdapter` — scripted `enqueue` per topic; registers consumers in a
  Map; deterministic `fake-msg-N` message ids.
- `FakeSecretsAdapter` — scripted by secret name; throws "not set" otherwise.
- `FakeFileRouterAdapter` — keyed by filename; returns a discriminated
  `RoutingDecision`.
- `FakePreprocessorAdapter` — keyed by mediaType; pass-through default.

## 3. Decisions

### D-1: Two sets of fakes coexist by design

**Why:** `dis/src/core/__fakes__/` already supplies single-shot fakes used
by the orchestrator test (one constructor arg = one canonical outcome).
DIS-012 requires a composable script-driven variant for core tests that
drive multiple inputs through the same fake. Merging them would force the
orchestrator to adopt the heavier API for no gain.
**How to apply:** orchestrator-level tests keep using `src/core/__fakes__/`;
new Wave-B core tests needing per-fixture branching import from
`tests/helpers/`.

### D-2: `FakeDatabaseAdapter` implements the full port, including named methods

**Why:** DIS-021b/d introduced domain-named methods on `DatabasePort` so
core stops embedding SQL. A structural fake that only implemented
`query`/`queryOne` would not typecheck against `DatabasePort` today.
**How to apply:** keep the named methods in lock-step with the port. If new
domain methods land (e.g. `findRevisions`), extend this fake in the same
commit.

### D-3: Imports stop at `src/ports/*` + one `src/core/state-machine.ts` type

**Why:** VERIFY-5 asserts no `src/adapters/*` imports. The State type is
a discriminated union of state names; it lives in core not ports but is
transitively referenced by `DatabasePort.updateExtractionStatus`. That
import stays within core/port territory and does not violate the
adapter-free rule.
