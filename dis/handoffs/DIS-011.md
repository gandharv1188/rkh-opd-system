### D-2: Bash validator is redundant-but-complementary to `fitness.mjs` — intentional

Wave-B session handover §6 notes that the DIP invariant is **already enforced** by `dis/scripts/fitness.mjs` rules `core_no_adapter_imports` + `ports_no_adapter_imports` (the legacy `port-validator.mjs` is a one-line wrapper over these). The DIS-011 ticket nonetheless calls for a `dis/scripts/port-validator.sh`. Decision:

- **Keep both.** The bash script is a shell-level tripwire that does not depend on the Node toolchain (no `tsc`/`ts-node`/`vitest` required to run). If the Node side ever regresses or fails to install, the bash check still runs in CI.
- **Do not remove fitness.** `fitness.mjs` covers seven rules; DIS-011 only asks for one. Removing it would lose coverage.
- **Wire the bash check into CI** — deferred to a follow-up (see F-1) rather than widening this ticket. VERIFY-4 already passes via the existing `port-validator` job that runs the `.mjs` wrapper.

### D-3: Grep pattern matches three import styles

The regex `from ['"][^'"]*(src/|\.\./|\./)adapters/` catches:

- Relative imports: `from '../adapters/foo.js'`, `from './adapters/foo.js'`
- Path-alias style: `from '@/src/adapters/foo.js'` (in case tsconfig paths are introduced later)
- Full-path style: `from 'src/adapters/foo.js'`

The tests currently pass with the tree clean so the regex is proved non-false-positive but not fully proved for every intended match; future DIS-011 follow-up could add a fixture-based negative test.

### D-4: Script is cwd-independent

Matches the pattern established by DIS-021c — resolves its own repo root via `BASH_SOURCE[0]` and `cd "$REPO"` so CI and local devs get identical behaviour regardless of cwd.

## 5. Follow-ups / flags

### F-1: Wire bash `port-validator.sh` into CI as a dedicated step

Currently the existing CI `port-validator` job invokes `node dis/scripts/port-validator.mjs`. For belt-and-braces coverage (per D-2) a second job or step could also invoke `bash dis/scripts/port-validator.sh`. Not blocking — file the follow-up as DIS-011-followup. Alternatively, the two could be consolidated by having the Node wrapper `spawnSync` the bash script, but that re-introduces the Node-toolchain dependency the bash script was meant to isolate from.

### F-2: Consider negative-path fixture test

Current test only exercises the happy path (clean tree). A follow-up could add a tmp-dir fixture containing a deliberate `src/core/bad.ts` that imports `../adapters/foo.js` and assert the script exits 1 with that line in stdout. Out of scope for DIS-011's file-level `files_allowed`.
