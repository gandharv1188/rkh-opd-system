## 1. What ships

The staging table every DIS document flows through, plus the generic
`idempotency_keys` table backing `dis/src/core/idempotency-store.ts` (DIS-025).

`ocr_extractions` is superset-shaped: it contains every column mentioned in
either `data_model.md` or the dispatch brief so the adapter code already
written (`dis/src/adapters/database/supabase-postgres.ts`) compiles against
the eventual schema without a follow-up. Columns unique to either source are
nullable / defaulted.

`idempotency_keys` matches the exact tuple the idempotency-store's
`SELECT_SQL` / `INSERT_SQL` fragments expect: `(key text PK, payload_hash
text NOT NULL, created_at timestamptz NOT NULL DEFAULT now())`.

## 2. Why M-001 owns `idempotency_keys` (not a separate M-001a)

The brief gave two options. I chose to include it in M-001 because:

- Both tables ship with the first DIS release — no later migration depends
  on `idempotency_keys` being absent.
- A separate M-001a would force an arbitrary dbmate ordering question
  (does it run before or after ocr_extractions?) with no semantic answer.
- `idempotency-store.ts` (DIS-025) references the table via parameterised
  literals only; bundling keeps the "one migration creates one concern" rule
  approximately intact (two concerns, one release).

If the team later wants them separate, splitting is trivial — no FKs cross
the pair.

## 3. Test plan

- `dis/tests/integration/migrations.test.ts` loads both SQL files as strings
  and asserts `create table ocr_extractions`, `create table idempotency_keys`,
  required indexes, and `gen_random_uuid()` usage. Rollback asserts the
  matching `drop table if exists`.
- Live forward → rollback → forward round-trip against an ephemeral Postgres
  is **deferred to Wave 7**: the test file contains a `maybeRunLiveRoundtrip`
  hook gated on `DIS_MIGRATIONS_LIVE` env that today no-ops. Static parse
  checks are sufficient per brief (Postgres may not be available in CI).

## 4. Dependencies

- None. This is the foundation migration; every later M-00X builds on it.
- Consumers (informational, not blockers):
  - `supabase-postgres.ts` already writes to `ocr_extractions` columns.
  - `idempotency-store.ts` already reads/writes `idempotency_keys`.

## 5. Gotchas

- `gen_random_uuid()` requires `pgcrypto`. The migration runs
  `create extension if not exists pgcrypto` so first-time POC setup works.
  Supabase enables pgcrypto by default; AWS RDS requires the extension to be
  in `rds.allowed_extensions` (check before Wave-7 prod application).
- `parent_extraction_id` references the same table (self-FK for retry
  lineage). The rollback uses `cascade` so this self-reference does not
  block the drop.
- `correlation_id` is NOT NULL; the app layer always generates one, so this
  is a deliberate invariant rather than a gotcha — callers will see
  `null value in column` if they forget.
