## 3. Decisions taken during implementation

### D-1: State-machine stub authored in DIS-021

DIS-020's branch is not merged into this worktree, so `./state-machine.js` would have been a dangling import. Rather than mock the module path inside the orchestrator, I created a minimal stub that exports the exact surface the orchestrator needs (`ExtractionState`, `ExtractionEvent`, `InvalidStateTransitionError`, `transition`). The transition table is a subset — only the edges the orchestrator uses. Reconciliation is the merger's responsibility.

### D-2: Idempotency payload hash is deterministic but non-cryptographic

`hashPayload()` concatenates patient id + filename + content type + body length. Crypto hashing belongs in an adapter (SHA-256 per coding_standards.md §6 A08). Keeping it out of core preserves DIP and keeps tests fast.

### D-3: Injected id generator + clock are optional

`OrchestratorDeps` exposes `now?` and `newId?` for deterministic tests. Falls back to an internal counter + `new Date()` when unset. The fake database tolerates both.

### D-4: `process()` drives both routing paths in one call

TDD §4's diagram shows preprocessing → ocr → structuring as three persisted states. `process()` advances the row through each hop explicitly via `advance()` so the audit log (DIS-024) sees one transition per edge. The fake DB exercises the optimistic-lock UPDATE on each hop.

### D-5: `retry()` uses `retry:<id>` as the idempotency key

Retries must not collide with the original idempotency key. Using `retry:<newId>` guarantees uniqueness while keeping the column NOT NULL.
