## 3. Decisions taken during implementation

### D-1: Port validator written as zero-dep Node ESM

**Context:** The CI step must run before any `npm ci` in case `dis/` is not yet bootstrapped on early branches.
**Options considered:** Node ESM with only `node:fs`/`node:path`; bash + grep; Python.
**Decision:** Node ESM (`.mjs`) with only built-in modules.
**Reason:** Keeps the script portable across Windows devs and Linux CI, no dependency on ripgrep/grep flag differences, runs before npm install.
**Revisit if:** the rule set grows beyond a handful of regex checks; at that point migrate to a typed project scanner.

### D-2: Workflow fails soft if `dis/package.json` is absent

**Context:** Early tickets (DIS-001..DIS-002) merge before the full `dis/` subproject exists.
**Options considered:** Require `dis/package.json` to exist; run lint/test conditionally.
**Decision:** Conditional step guards (`if: hashFiles('dis/package.json') != ''`).
**Reason:** Otherwise DIS-002 itself cannot pass its own CI on this branch.
**Revisit if:** DIS-001 lands and the guard becomes dead code — simplify it then.

### D-3: Gitleaks as the secret scanner

**Context:** AC-4 specifies gitleaks by name.
**Options considered:** trufflehog, gitleaks, detect-secrets.
**Decision:** `gitleaks/gitleaks-action@v2` at default config.
**Reason:** AC-4 is explicit. Default ruleset is strong enough for POC; custom `.gitleaks.toml` can come later.
**Revisit if:** false-positive rate becomes a CI nuisance.
