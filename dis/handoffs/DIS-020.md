## 3. Decisions taken during implementation

### D-1: Per-event `switch` arms rather than a transition lookup table

**Context:** Two common patterns — a nested `Record<State, Record<EventKind, State>>` table, or a `switch` over event kinds with inner `if` guards on current state.
**Options considered:** (a) record-lookup with `??` throw; (b) switch over event with state guards.
**Decision:** (b). Switch over `event.kind` with `assertNever(event)` default.
**Reason:** Exhaustiveness of events is the safety-critical invariant (every event must be classified); coding_standards §1 explicitly mandates `assertNever` in the discriminator switch. State-arity checks inside each case are trivial and keep the payload narrowing (e.g., `event.reason`) in scope.
**Revisit if:** the table grows beyond ~15 events — then an explicit data-driven table with a single throw path may be clearer.

### D-2: `upload` event is always invalid

**Context:** TDD §4 shows `uploaded` as the entry state, not a transition _into_ `uploaded`. An orchestrator creates the row in `uploaded` directly.
**Decision:** `upload` event is included in the `Event` union for symmetry with the orchestrator vocabulary, but the transition function always throws — the state machine cannot synthesize the initial row.
**Revisit if:** DIS-021 wants an explicit `from: null` entry edge; we can add a `'initial'` pseudo-state then.

### D-3: `fail` is a domain event, not a thrown error

**Context:** Adapters failing (e.g., OCR provider down) must land the extraction in `failed`, not bubble as uncaught.
**Decision:** Model `fail` as a first-class `Event` with a `reason` payload. Valid from `preprocessing`, `ocr`, `structuring`, `uploaded`.
**Revisit if:** retry semantics need a distinct `retry_scheduled` state.
