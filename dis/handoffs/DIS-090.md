## 3. Decisions taken during implementation

### D-1: Size check AFTER reading arrayBuffer

**Context:** The brief allows either pre-parse content-length rejection or post-read length check. Hono's Request abstraction doesn't expose a streaming cutoff cleanly.
**Options considered:** (a) inspect `content-length` header first, (b) read full body then measure.
**Chosen:** (b), per brief ("simpler, matches POC scope"). Trade-off: caller still consumes bandwidth for oversize upload, acceptable for POC. Future hardening (DIS-096 signed uploads) removes this path entirely.

### D-2: `c.get('correlationId' as never)` cast

**Context:** `registerIngestRoute` takes plain `Hono` (matching admin-metrics/health convention) — not `Hono<{Variables: AppVariables}>` — so `c.get('correlationId')` errors with TS2769.
**Options considered:** (a) widen signature to typed Hono, (b) cast the key.
**Chosen:** (b). Rationale: keeps route registration API consistent with siblings. `server.ts` bolts the route onto a typed app; the middleware chain owns the variable. Same pattern will likely appear in siblings 091-094. Documented the trick inline.

### D-3: Content-type parameter stripping

**Context:** Real browsers send `application/pdf; charset=binary` etc.
**Chosen:** Split on `;` and trim the first segment before allow-list lookup. Matches test case 9.

### D-4: FakeOrchestrator cast via `unknown`

**Context:** `IngestionOrchestrator` has many private fields and port deps. A structural fake that only implements `ingest()` can't satisfy that via declaration-merging.
**Chosen:** `fake as unknown as IngestionOrchestrator`. Standard test-only escape valve; only the `ingest()` method is exercised.
