# DIS-092 — POST /extractions/:id/approve

## 1. Scope
Implements `POST /extractions/:id/approve` — nurse/doctor approval of a
verified extraction under optimistic lock. Calls `orchestrator.approve()`
which routes the `nurse_approve` event through the state machine
(CS-1) and (on follow-up) triggers promotion with DB-backed dedupe
(CS-10).

## 3. Design decisions
- **Handler stays dumb.** It only validates body + actor and forwards to
  `orchestrator.approve()`. No state-machine shortcuts, no dedupe logic.
  This preserves CS-1 (state machine is the single authority for
  transitions) and CS-10 (promotion dedupe is owned by DB unique indexes
  from M-007).
- **Error surface is DIS-101's handler.** `VersionConflictError` and
  `ExtractionNotFoundError` propagate from the orchestrator and are
  mapped to 409 / 404 envelopes by `errorHandler()`. The test app wires
  `app.onError(errorHandler())` and asserts the envelope — this is the
  integration surface we care about.
- **`promotion.inserted`/`skipped` are placeholders.** `approve()`
  currently returns `ExtractionRecord`. A follow-up ticket should widen
  the return to include the promotion summary (inserted / skipped
  counts). Documented inline in the route file and the original brief.
- **Actor precedence.** Body `actor` wins, then `X-Operator-Id` header;
  neither present → 400 `MISSING_ACTOR`.

## 4. CS sign-off (batched per squad policy — final review before Epic G)
- **CS-1 (no state-machine bypass):** handler delegates all state
  changes to `orchestrator.approve()`. No direct DB writes, no
  transition() calls. Any attempt to bypass would still be rejected by
  the state machine.
- **CS-10 (promotion dedupe):** not implemented in the handler by
  design — dedupe is enforced at the DB level (M-007 unique indexes
  on `(patient_id, canonical_hash)` etc.). Handler surfaces whatever
  the orchestrator returns.
