## 1. What ships

`rateLimit(config?)` — a Hono middleware that enforces a token-bucket
rate limit keyed on the `X-Operator-Id` request header (header name
configurable). Defaults: 60 tokens per 10-minute window. Storage is an
in-process `Map<operatorId, Bucket>`; POC only.

Signature:

```ts
export interface RateLimitConfig {
  readonly maxTokens?: number;            // default 60
  readonly refillPerMs?: number;          // default maxTokens / windowMs
  readonly windowMs?: number;             // default 600_000
  readonly now?: () => number;            // injectable clock (default Date.now)
  readonly operatorIdFromHeader?: string; // default 'X-Operator-Id'
}

export function rateLimit(config?: RateLimitConfig): MiddlewareHandler;
```

## 2. Behaviour

- **Missing header** → 400 `MISSING_OPERATOR_ID`, `retryable: false`.
- **Under limit** → passes to `next()`, consuming one token.
- **Exhausted** → 429 `RATE_LIMITED`, `retryable: true`, `Retry-After`
  integer seconds header + `details.retry_after_seconds` in envelope.
- **Refill** → continuous: `tokens += elapsed * refillPerMs`, clamped to
  `maxTokens` on every request.
- **Per-operator isolation** → separate bucket entries keyed by operator
  id; one operator's exhaustion never blocks another.

Error envelope matches the canonical
`{error:{code,message,retryable,correlation_id,details?}}` shape used by
DIS-018 / DIS-101, pulling `correlation_id` from the Hono context
(empty string when not set upstream, which the test harness populates).

## 3. Why clock injection

All 7 tests use a mutable `clock` variable exposed via `now: () => clock`
to the middleware. That lets "bucket refills over time" assert behaviour
after advancing `clock += 600_000` without any `setTimeout`. No
real-time sleeps anywhere → zero flakiness, sub-20 ms suite.

## 7. Out of scope (intentional)

- **Server registration.** No edit to `dis/src/http/server.ts` — orchestrator
  wires middleware order globally when it stitches DIS-100/101/102 together.
- **Redis-backed storage.** In-process Map only. A production port that
  swaps the `Map` for a Redis KV behind a `RateLimiterStore` interface is
  a Phase-2 ticket (likely DIS-2xx once multi-instance deploys land).
- **E2E rate-limit integration.** Deferred to DIS-110 (full-stack HTTP
  harness covers all middlewares composed with real routes).
- **Per-route overrides / weight-based consumption.** Not required by
  TDD §6; trivial to add (accept a `cost` arg on `next()`) if/when needed.

## 8. Risks / follow-ups

- **Map grows unbounded** across operator ids. For a POC with O(100)
  operators this is fine; production port should add either a LRU cap
  or a janitor that drops buckets idle > `windowMs`.
- **Clock source.** Uses `Date.now` by default, which is wall-clock and
  can drift under NTP adjustments. Acceptable for rate-limit windows.
  A future port could prefer `performance.now()`-derived monotonic time
  if drift is observed.

The in-process `Map` is the single line that does not graduate to prod
unchanged. Everything else — the token-bucket math, envelope shape,
`Retry-After` formatting, injectable clock, header configurability —
is production-shaped and should carry forward verbatim.
