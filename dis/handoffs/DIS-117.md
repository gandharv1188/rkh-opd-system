## 3. Decisions taken during implementation

### D-1: Track viewed-state with `Set<string>`, key by field name

**Context:** CS-1 requires "every field reviewed before approve." Needed a way to observe per-field interaction.
**Options considered:** (a) `onFocus` + `Set`, (b) `onClick` + `Array`, (c) IntersectionObserver on each row.
**Decision:** (a).
**Reason:** Focus is a deliberate user action (keyboard or click through tabIndex=0), aligns with a11y, and a `Set` is O(1) for the `every` check on each render.
**Revisit if:** Product wants "viewed" to mean "visibly dwelt on for N ms" rather than "focused once" — would switch to IntersectionObserver + timer.

### D-2: Test asserts shell, not component behaviour

**Context:** Brief supplied a spec stub explicitly calling out that component-level tests require routing that isn't wired yet.
**Options considered:** (a) ship spec as stub per brief, (b) wire up a minimal `/verify/:id` route in `App.tsx` to exercise the component.
**Decision:** (a).
**Reason:** Brief STRICTLY limits writable files to `Verify.tsx`, `verify.spec.ts`, and this handoff. Touching `App.tsx` is out of scope for DIS-117; orchestrator wants the follow-up tracked as DIS-117-followup.
**Revisit if:** Sibling ticket wires routing — then this test upgrades to assert `data-testid="approve-button"` is disabled on load, focus each `field-*` in turn, assert enabled.

### D-3: `iframe` for PDF rather than reusing `PdfViewer` component

**Context:** DIS-118 shipped a PDF.js viewer component. Could have composed it in.
**Options considered:** (a) plain `<iframe>`, (b) import `PdfViewer` from DIS-118.
**Decision:** (a), matching the brief's supplied implementation verbatim.
**Reason:** Brief provided the exact implementation to use. Swapping to the component is a composition concern for the routing/integration ticket.
**Revisit if:** The routing ticket composes — it should replace the `<iframe>` with `<PdfViewer url={detail.pdf_url} />`.
