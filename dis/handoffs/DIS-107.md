# DIS-107 — E2E version conflict on approve

## Scope

End-to-end HTTP-level regression guard for the optimistic-lock contract
at the approve seam. Two operators approve the same extraction
concurrently: the first wins (version 1 → 2), the second arrives with a
now-stale `expected_version=1` and must receive
`409 VERSION_CONFLICT` through the DIS-101 error envelope.

Covers the full stack: route body parsing (DIS-092), orchestrator
optimistic lock (DIS-026), `VersionConflictError` → envelope mapping
(DIS-101) — all wired through `createServer()`.

## Files

- `dis/tests/e2e/version-conflict.test.ts` (new) — single-spec Vitest
  E2E that builds an in-process Hono app via `createServer(...)`,
  drives two sequential `POST /extractions/:id/approve` requests, and
  asserts HTTP 409 plus `body.error.code === 'VERSION_CONFLICT'`.

## Test design

- In-memory `DatabasePort` fake (inline): seeds one row in
  `ready_for_review`, version=1; `updateExtractionStatus` implements
  compare-and-set on version. Only the three methods touched by the
  approve path are implemented; the rest throw "unused in this test".
- Adapter ports the approve path never touches
  (storage/queue/secrets/fileRouter/preprocessor/ocr/structuring) are
  passed as throwing Proxy stubs — any future coupling to them would
  be caught immediately.
- Uses `app.request(...)` (Hono's in-process dispatcher) — no ports,
  no sockets, keeps the test hermetic and fast.
