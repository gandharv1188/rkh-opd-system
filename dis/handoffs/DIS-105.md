# DIS-105 — E2E reject path

## 1. Ticket
E2E: operator rejects an extraction; assert no row inserted into `lab_results`.

## 2. Branch / worktree
- Worktree: `.claude/worktrees/dev-d-e2e-reject`
- Branch: `feat/dev-d-e2e-reject`

## 3. Files changed
- `dis/tests/e2e/reject.test.ts` (new)
- `dis/handoffs/DIS-105.md` (new)

## 4. Approach
Composed the real HTTP surface for the paths exercised by the flow:
`POST /ingest` and `POST /extractions/:id/reject`, wired to a real
`IngestionOrchestrator` backed by the shared `Fake*Adapter` fleet. The test
drives ingest (native-text route) → `orchestrator.process()` →
reject, then asserts the reject produced no side-effects on `lab_results`.

`process()` has no HTTP route (it is worker-driven in the TDD), so the
test calls it directly on the orchestrator after the HTTP ingest. The
reject step is exercised through the HTTP surface exactly as the brief
specified.

## 5. Assertion chosen (and why)
Two complementary assertions:

**A. `labInsertCalls.length === 0`** — wrapped `FakeDatabaseAdapter.query()`
to capture any call whose SQL contains `insert into lab_results`. This is
the brief's preferred spy-style assertion. Today it is vacuously satisfied
because promotion is not yet wired into the orchestrator
(`extractions-approve.ts:18` calls out the placeholder); the spy stays in
place so that a future promotion wire-up that leaks through reject will
correctly fail this test.

**B. State never enters `promoted`** — the extraction ends in `rejected`,
and no `updateExtractionStatus` call in `db.calls` carries
`newStatus === 'promoted'`. This is the brief's alternative "approve was
never called / promoted never entered" assertion. The state machine only
admits `promoted` from `verified` or `auto_approved`, so the presence of
this assertion pins the invariant: reject is a terminal fork.

Taken together: reject has zero side-effects on `lab_results` both at the
SQL layer (A) and at the state-transition layer (B).
