## 3. Decisions

### D-1: Loader does not validate schema

**Why:** validation is a consumer concern — Ajv compilation is
per-consumer, and the loader must stay useful for non-schema fixtures
(e.g. raw OCR responses). The test suite validates explicitly via Ajv to
prove the committed fixture is schema-valid.
**How to apply:** any future fixture of a schema-backed shape should get a
companion Ajv check in its own test.

### D-2: Ajv loaded via `createRequire`

**Why:** Ajv ships with a CJS default export + named `default` that trips
NodeNext's ESM-interop typing (`TS2351: not constructable`). Using
`createRequire('ajv')` gives an unambiguous CJS function reference with
a small local type.
**How to apply:** reuse the same pattern when pulling other CJS packages
into ESM-strict TS tests.
