## 3. Decisions taken during implementation

### D-1: In-memory `Map` keyed by `name{labels}`, not Prometheus histograms

**Context:** ticket scope is a POC stub. Real exposition format + histograms land in DIS-148.
**Decision:** counters only; gauges stubbed as empty in the snapshot for forward-compat.
**Revisit if:** DIS-148 starts; expand this port before wiring real adapter.

### D-2: Inject `metrics` into the route, default to the process singleton

**Context:** tests must assert counter math without polluting the shared singleton.
**Decision:** `registerAdminMetricsRoute(app, metrics = getDefaultMetrics())`.
**Revisit if:** we add multiple scoped registries (e.g. per-tenant) — unlikely soon.

### D-3: No auth on `/admin/metrics`

**Context:** ticket explicitly lists auth as out of scope.
**Decision:** leave open; document for ops.
**Revisit if:** deploying beyond POC — before that, block via reverse proxy.

## 4. What was deliberately NOT done

- No Prometheus text format; no OpenTelemetry; no histograms (DIS-148).
- No wiring of the singleton into the orchestrator; adapters will `inc()` as they're instrumented in follow-ups.
- No auth middleware (explicit ticket out-of-scope).

## 10. Non-obvious gotchas

- Label equality is position-independent: `{a=1,b=2}` and `{b=2,a=1}` map to the same series because `keyFor()` sorts keys.
- The current snapshot exposes `gauges: []` even though gauges are not yet implemented; consumers (including DIS-148) should assume the shape is stable.
- The singleton from `getDefaultMetrics()` is truly process-wide; tests that touch it must `reset()` or use `createMetrics()` to avoid cross-suite bleed.
