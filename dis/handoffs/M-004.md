## 1. What ships

Lightweight job queue backing `PgCronQueueAdapter` (DIS-056, in flight in
sibling worktree). Workers pull rows where `completed_at IS NULL`, partition
by `topic`, and respect `available_at` for delayed execution.

Partial index `idx_dis_jobs_ready` on `(topic, available_at) WHERE
completed_at IS NULL` keeps scans tight even as completed rows accumulate.

## 2. Env gating — handled by caller, not SQL

dbmate has no primitive for "skip this file when env X". Options considered:

1. Wrap the body in `DO $$ BEGIN IF current_setting('dis.stack') = 'aws'
   THEN RETURN; END IF; ... END $$;` — works, but adds a migration-time
   config knob and obscures what the migration does.
2. Filter the file out at the wrapper layer (`migrate.sh` inspects
   `DIS_STACK` and passes `--exclude` or a filtered dir).
3. Apply always — Postgres is always available on AWS too (RDS).

Chose option **3 with a header comment** because: this table is harmless
on AWS (it just sits empty — SQS is used instead, the adapter won't touch
it), and the "AWS skip" requirement is weaker than the "portable SQL"
requirement. If Epic H later wants it gone on AWS, option 2 is a one-line
wrapper change.

## 5. Gotchas

- `scheduled_for` vs `available_at` — both exist to accept either column
  name the caller emits. They default to `now()` so a caller that sets
  neither still works; a caller that sets one should set both to the same
  value until the codebase picks a single convention.
- Index is `(topic, available_at)` — not just `(topic)` — so SELECTs using
  `WHERE topic = $1 AND available_at <= now()` stay index-only.
