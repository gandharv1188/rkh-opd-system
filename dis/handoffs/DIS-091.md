## 1. What ships

`registerExtractionsGetRoute(app, { db })` — a Hono route factory that
registers `GET /extractions/:id`. Injects a `DatabasePort`; no module-level
singletons. Returns the persisted extraction row projection for the
verification UI; 404s on absence.

Signature:

```ts
export interface ExtractionsGetRouteDeps { readonly db: DatabasePort; }
export function registerExtractionsGetRoute(
  app: Hono<{ Variables: AppVariables }>,
  deps: ExtractionsGetRouteDeps,
): void;
```

## 2. Behaviour

- **Row found** → 200 JSON with `{ id, patient_id, status, version, parent_extraction_id }`.
- **Row not found** → 404 `EXTRACTION_NOT_FOUND`, `retryable: false`, via the
  canonical error envelope from `errorEnvelope()` / `HttpError`.
- **Cross-patient access (RLS)** → adapter enforces RLS via session vars
  (`app.role`, `app.patient_id`); denied rows surface as `null` from
  `findExtractionById`, which the route maps to the same 404. Intentional —
  do not leak existence.

## 3. Gap: projected fields vs. backlog VERIFY-2

VERIFY-2 names six fields that the verification UI will eventually need:
`raw_ocr_markdown`, `raw_ocr_blocks`, `structured`, `verified_structured`,
`confidence_summary`, `version`. Of those, **only `version` is on the
current `ExtractionRow`** (port in `dis/src/ports/database.ts`). The
remaining five are persisted today as columns on the `extractions` table
but are not exposed through the `DatabasePort` surface.

**What this route does now:** projects the full set of *available* fields
from `ExtractionRow` — `id`, `patient_id`, `status`, `version`,
`parent_extraction_id` — plus the verification UI contract stub shape.

**Proposed follow-up (DIS-091a):** extend `DatabasePort` with
`findExtractionForVerification(id) → ExtractionVerificationView | null`
whose return type adds the five deferred fields. The route then maps 1:1
to the extended view without changing its public contract (status + code
unchanged).

I kept the VERIFY-2 test title verbatim per brief so the orchestrator
can grep for it, and documented the field gap here rather than in the
test, because the test asserts the *current* projection (not the
aspirational one).

## 8. Risks / follow-ups

- **Field gap (§3)** — the verification UI will hit this endpoint
  expecting richer content. Surface DIS-091a before UI wiring, or the
  UI will receive a thinner payload than its type definition claims.
- **Existence leak** — confirmed none: 404 envelope body contains only
  the opaque id that the caller supplied. No timing side-channel mitigation
  (both found/not-found paths are a single DB round-trip).

## 10. POC vs. production boundary

The route handler itself is production-shaped — pure dependency
injection, no module state, RLS-aware through the port contract. The
one line that changes in production is the DB adapter that backs
`findExtractionById`: the Supabase POC uses PostgREST under the hood,
the AWS deployment will hit RDS via `pg`. Both must honour
`setSessionVars` so the RLS behaviour tested here holds end-to-end.
