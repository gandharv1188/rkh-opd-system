This is the **direct fix for orientation finding F1**: prior to this ticket
`dis/src/wiring/` was empty and the service could not boot end-to-end.
Every adapter, port, and core component built so far now composes through
a single entry point.

## 1. What ships

`createSupabasePorts(source?)` returns a typed `Ports` bag:

```ts
type Ports = {
  database: DatabasePort;
  storage: StoragePort;
  ocr: OcrPort;
  structuring: StructuringPort;
  fileRouter: FileRouterPort;
  preprocessor: PreprocessorPort;
  secrets: SecretsPort;
  queue?: QueuePort; // undefined until DIS-056 lands
};
```

Plus helpers:

- `composeForHttp(ports): Hono` — wraps `createServer()`; route tickets
  add `ports` into the created app once they ship.
- `bootSupabase(source?)` — one-call convenience returning `{ ports, app }`.
- `__setDriverLoaderHookForTests` / `__resetDriverLoaderHookForTests` —
  internal test seams for the postgres driver loader; NOT for production.
- `createAwsPorts()` in `aws.ts` throws `"AWS wiring not yet implemented —
  see Epic H"` per ADR-001.

## 3. Decisions (per brief: every coupling + every sibling NOT wired)

### 3.1 SecretsPort is a local env-backed shim, NOT DIS-055

DIS-055 (`SupabaseSecretsAdapter`) is in flight in a sibling worktree. The
adapters present today (`DatalabChandraAdapter`, `ClaudeHaikuAdapter`,
`SupabaseStorageAdapter`) all accept a `SecretsPort` by DI, so the
composition root supplies `EnvSecretsAdapter` — a 12-line shim whose `get()`
reads from the validated `Env` object (and so accepts the same names
DIS-055 will: `DATALAB_API_KEY`, `ANTHROPIC_API_KEY`, `SUPABASE_URL`,
`SUPABASE_SERVICE_ROLE_KEY`).

**When DIS-055 lands:** swap one line in `createSupabasePorts`
(`const secrets = new SupabaseSecretsAdapter(...)`); no adapter call site
changes. The `EnvSecretsAdapter` class can be deleted in the same PR.

### 3.2 Postgres driver loader — test-overridable, production-default

`SupabasePostgresAdapter` uses a `setPostgresDriverLoader(loader)` seam
(ADR-006). The composition root registers `defaultDriverLoader` which
dynamically calls `postgres(connectionString)` from the `postgres` npm
package. For integration tests, `__setDriverLoaderHookForTests` routes the
registration onto a fake `SqlClient` so the suite does not need a live
Postgres (or even a resolvable `postgres` module).

**Test observation:** the loader is invoked *exactly once per adapter
construction* (eagerly, in `SupabasePostgresAdapter` constructor via
`createDefaultSqlClient`). The test asserts `loaderCalls === 1` after one
`createSupabasePorts` call — this pins the contract "the wiring does not
accidentally double-construct the DB client".

### 3.3 Siblings NOT wired (deliberate — not visible on this baseline)

These adapters are in flight in sibling Wave-3a worktrees; none were on
`feat/dis-plan` at dispatch, so this PR does NOT import them. A post-merge
follow-up wires them in with a single edit each:

| Ticket      | Port              | Sibling worktree             | Follow-up action                                       |
| ----------- | ----------------- | ---------------------------- | ------------------------------------------------------ |
| DIS-052     | `OcrPort`         | `dev-c-claude-vision`        | Conditional: use if `env.DIS_OCR_PROVIDER === 'claude'` |
| DIS-055     | `SecretsPort`     | `dev-c-missing-adapters`     | Replace `EnvSecretsAdapter` (see §3.1)                  |
| DIS-056     | `QueuePort`       | `dev-c-missing-adapters`     | Set `Ports.queue = new PgCronQueueAdapter(...)`         |
| DIS-058a..c | `PreprocessorPort` | `dev-c-preprocessor-pipeline` | Replace `DefaultPreprocessor` with real pipeline        |
| DIS-059     | Native PDF        | `dev-c-missing-adapters`     | `DefaultFileRouter` already consumes `pdfjs-dist` via default loader; no wiring change expected unless a named adapter lands |
| DIS-060     | Office Word       | `dev-c-office-parsers`       | New route handler inside orchestrator, no wiring change  |
| DIS-061     | Office Sheet      | `dev-c-office-parsers`       | Same as DIS-060                                        |

Full Wave-3a merge protocol: after all siblings land on `feat/dis-plan`,
one follow-up ticket edits `createSupabasePorts` to pick the correct
adapter instances based on `env.DIS_OCR_PROVIDER`, `env.DIS_STRUCTURING_PROVIDER`,
and the presence of DIS-055/056.

### 3.4 DATALAB_API_KEY flow

Brief said "read from env directly for now; document the swap in §5."
Better option chosen: `DatalabChandraAdapter` already takes a
`SecretsPort`, and the env-backed shim implements it. So the key flows
through the port today, which means when DIS-055 lands **nothing changes
for Datalab** — the port wiring stays identical. Net effect: no swap
needed for Datalab when DIS-055 merges; only the `secrets` field in the
bag is replaced.

### 3.5 `composeForHttp` is a placeholder boundary

`createServer()` in `dis/src/http/server.ts` does not yet accept ports.
`composeForHttp(ports)` is intentionally a no-op parameter consumer so
that:

- DIS-004 (upload route) only needs to change `composeForHttp` signature
  — it does not need a second wiring PR.
- The test at `wiring-supabase.test.ts` can round-trip `/health` today to
  prove the bag ↔ HTTP seam is wired.

The `_ports` parameter in `composeForHttp` is prefixed with `_` and
documented as "do not remove"; it is part of the public contract.

### 3.6 AWS stub throws early

Per ADR-001 / `portability.md`: `createAwsPorts` is a single-line throw.
Listing the file gives Epic H a target to fill without a broader PR.

### 3.7 File placement diverges from backlog

Backlog `files_allowed`: `dis/src/wiring/poc.ts` + `dis/tests/wiring/poc.test.ts`.
Brief specifies: `dis/src/wiring/supabase.ts` + `dis/tests/integration/wiring-supabase.test.ts`.

Chose the brief's paths because (a) `supabase`/`aws` names align with the
`DIS_STACK` enum from `env.schema.ts`, making the selector one line in a
future `createPortsForStack(env.DIS_STACK)`; (b) integration test
placement matches existing `tests/integration/` convention.

Follow-up DIS-079-followup-a can rename to `poc.ts` if the team prefers
the backlog naming.
