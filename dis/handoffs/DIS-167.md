# DIS-167 — Chaos: DB drop mid-txn

## Summary
Simulated a Postgres connection drop in the middle of a transaction and verified
that no partial write persists — a cornerstone invariant for the extraction
state machine and `lab_results` ingest path.

## Files added
- `dis/tests/chaos/db-drop.test.ts` — vitest chaos test.

## Approach
- Hand-rolled fake `DatabasePort` (`makeFlakyDb`) that throws
  `ConnectionDroppedError` on the Nth `query()` inside a `transaction()`.
- `transaction()` uses a scratch buffer; the in-memory store only receives
  the scratch writes if `work()` completes without throwing. When the drop
  fires mid-work, the scratch is discarded via the `finally` path — exactly
  the semantic real Postgres gives us on ROLLBACK.
- Test arranges 4 INSERTs with a drop configured after 3. The 3rd query
  throws; the 4th never runs; assertions confirm:
  - Error propagates to caller.
  - `__rows()` is empty (no commit).
  - `__committed()` is `false`.
