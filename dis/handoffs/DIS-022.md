## 3. Decisions taken during implementation

### D-1: Missing field confidence treated as 0

**Context:** Policy may list a field that the structured extraction didn't populate.
**Decision:** `structured.fields[rule.field]?.confidence ?? 0`.
**Reason:** Fail-closed (coding_standards §2); silently passing absent fields would violate CS-7.
**Revisit if:** a field legitimately means "not applicable" and should be skipped — then policy should express that with an explicit rule shape.

### D-2: `block_type` constraint modelled as rule-level optional field

**Context:** TDD §12 example uses an expression `confidence >= 0.95 AND block_type = 'table'`. Parsing arbitrary DSL expressions is out of scope for the pure evaluator.
**Decision:** Structured `ConfidenceRule` with `threshold: number` and optional `block_type`. Loader/adapter is responsible for compiling the stored JSON expression into this structured form.
**Reason:** Keeps the core evaluator trivially unit-testable and portable; expression parsing belongs in the adapter layer.
**Revisit if:** policy authors need richer composed predicates (`AND`/`OR` across multiple block attributes).

### D-3: Empty `rules` ⇒ `auto_approved=false` when enabled

**Context:** A misconfigured enabled policy with zero rules must not silently approve everything.
**Decision:** `rule_results.length > 0 && rule_results.every(passed)`.
**Reason:** Fail-closed; an empty policy is treated as "no rule matched".
