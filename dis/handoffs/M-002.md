## 1. What ships

Append-only audit log for DIS. Every state transition, field edit,
approve/reject, retry, and override is recorded by the orchestrator and
never mutated again. `BEFORE UPDATE` and `BEFORE DELETE` triggers that call
`ocr_audit_log_immutable()` raise an exception ‚Äî making the invariant survive
even a misconfigured service role or a bad RLS policy.

Schema is superset-shaped to accept both the brief's column names (`event`,
`actor`, `subject_id`, `before`, `after`) and `data_model.md`'s names
(`event_type`, `actor_type`, `actor_id`, `from_state`, `to_state`,
`field_path`, `before_value`, `after_value`). Duplicates are nullable so
callers can use whichever pair they prefer; a future migration can collapse
once the orchestrator's audit writes standardise on one shape.

## 2. Decisions

- **plpgsql trigger, not a RULE.** Rules can be bypassed with `INSERT ... ON
  CONFLICT DO UPDATE`; a `BEFORE UPDATE` trigger cannot.
- **Function drop ordered before table drop** in rollback: Postgres requires
  the trigger to be dropped first (or `CASCADE`), then the function.
- **No DELETE policy in M-008 either** ‚Äî defence in depth. Even with table
  owner privs, `DELETE` triggers this exception.

## 3. Test plan

- Static parse checks in `migrations.test.ts` ‚Äî asserts both triggers exist

  invariant test asserts M-002 includes both `before update on ocr_audit_log`
  AND `before delete on ocr_audit_log`.
- Live assertion (Wave 7): forward ‚Üí insert a row ‚Üí try UPDATE ‚Üí expect
  exception ‚Üí try DELETE ‚Üí expect exception ‚Üí rollback ‚Üí forward again.

## 4. Dependencies

- FK on `ocr_extractions(id)` ‚Äî M-001 must be applied first.

## 5. Gotchas

- `BEFORE UPDATE` fires even when no columns actually change (`UPDATE ...
  SET x = x`). This is intentional: it guarantees the invariant regardless
  of subtle SQL.
- If a future migration needs to fix a bad audit row, it must drop both
  triggers in a transaction, do the correction, and re-create the triggers
  ‚Äî with full incident documentation. Do not add an "admin bypass" column;
  that would defeat CS-3.

## 6. Follow-ups

- Once orchestrator audit writes settle on a single column set
  (likely `data_model.md`'s `event_type`/`actor_type`/`before_value`/
  `after_value`), deprecate the `event`/`actor`/`before`/`after` aliases.
