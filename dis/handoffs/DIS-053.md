## 1. What was built

- `dis/src/adapters/storage/supabase-storage.ts` — `SupabaseStorageAdapter` implementing `StoragePort` (5 methods: `putObject`, `getObject`, `getSignedUploadUrl`, `getSignedDownloadUrl`, `deleteObject`). Uses Node `fetch` against Supabase Storage REST (no `@supabase/supabase-js` dependency — keeps the adapter thin, portable, and easily swappable for an `S3Adapter`). Typed errors: `ObjectNotFoundError` (404), `StorageProviderError` (non-OK non-404). URL-encodes each path segment while preserving `/` separators. Credentials (`SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`) resolved via injected `SecretsPort`.
- `dis/src/adapters/storage/__fakes__/supabase-storage.ts` — `FakeSecrets` + `createFetchMock` helpers (scripted responses keyed by `${method} ${url}`, records every call).
- `dis/tests/unit/adapters/supabase-storage.test.ts` — 9 unit tests covering happy paths, 404 → `ObjectNotFoundError`, 5xx → `StorageProviderError`, URL encoding of spaces, and credential wiring from `SecretsPort`.

Layering: adapter imports only from `../../ports/storage.js` and `../../ports/secrets.js`. No core ↔ adapter leaks.

## 2. Endpoints used (Supabase Storage REST)

| Op                   | Method | Path                                            |
| -------------------- | ------ | ----------------------------------------------- |
| putObject            | POST   | `/storage/v1/object/{bucket}/{key}`             |
| getObject            | GET    | `/storage/v1/object/{bucket}/{key}`             |
| getSignedUploadUrl   | POST   | `/storage/v1/object/upload/sign/{bucket}/{key}` |
| getSignedDownloadUrl | POST   | `/storage/v1/object/sign/{bucket}/{key}`        |
| deleteObject         | DELETE | `/storage/v1/object/{bucket}/{key}`             |

All requests carry `Authorization: Bearer <SUPABASE_SERVICE_ROLE_KEY>`.
