## 4. Design notes
The test composes `createServer(...)` with in-memory fakes and drives the full pipeline through `app.fetch(...)` (no bound socket, parallel-safe). A bespoke `FlakyOcrAdapter` throws on its first call and returns a valid `OcrResult` on the second — the minimum needed to script transient-failure-then-recovery across two extraction rows.

Routes wired: `ingest`, `extractionsGet`, `extractionsRetry`, `processJob` (worker-token-guarded). All other composition root knobs default off.

## 5. State-machine expectations asserted
- id1 after first `process-job`: HTTP 500 (orchestrator rethrows post-fail transition).
- id1 after retry: row still present, `status !== 'uploaded'` (persisted `failed` transition), `parent_extraction_id === null`.
- id2 after retry creation: `status === 'uploaded'`, `version === 1`, `parent_extraction_id === id1`.
- id2 after second `process-job`: `status !== 'uploaded'` (advanced through full pipeline to `ready_for_review`).

## 6. Decisions / tradeoffs
- Asserted `status !== 'uploaded'` rather than an exact terminal status. The fitness rule + ports contracts pin exact statuses in unit tests; here we only need to prove "advanced". This keeps the test resilient to harmless renames of the ready-for-review state.
- Did not add a separate `/extractions/:id` parent-vs-child comparison of `version` values; FakeDatabaseAdapter's version arithmetic is already covered by `orchestrator-retry.test.ts` (DIS-035). This test guards the HTTP-level composition.
- Used a hand-written `FlakyOcrAdapter` instead of extending `FakeOcrAdapter`. The script-keyed fake only supports a single result per key, and the orchestrator emits the same synthetic key on both calls. A counter-based wrapper is clearer than stuffing fail-then-succeed into the script map.
