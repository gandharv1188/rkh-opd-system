## 1. What ships

Append-only ledger of DIS spend. One row per provider call (ocr or
structuring), linked to the extraction that triggered it. Cost stored as
`bigint` microrupees (1 INR = 1,000,000 micros) so aggregation is exact and
never drifts into float.

Same append-only trigger pattern as M-002 — `dis_cost_ledger_immutable()`
raises on UPDATE/DELETE.

## 2. Decisions

- **FK `ON DELETE SET NULL`, not RESTRICT.** Extractions are never deleted
  per CS-2, but if a future operational incident ever needs one gone, the
  ledger should retain its rows for finance (the extraction_id just
  becomes null). data_model.md mandates this — took that over the brief's
  unspecified default.
- **Both `input_tokens/output_tokens` and `tokens_in/tokens_out`** columns
  — again, superset to absorb both naming conventions in flight. Aggregation
  queries should `coalesce(input_tokens, tokens_in, 0)`.

## 3. Test plan

- Static parse — asserts table, both triggers, `raise exception`.
- Invariant test — asserts both `before update` AND `before delete`
  triggers exist.
- Live (Wave 7): insert row → attempt update → expect exception.

## 4. Dependencies

- FK on `ocr_extractions(id)` — M-001 first.

## 5. Gotchas

- Integer microrupees means a 100 INR cost is `100_000_000` — big numbers
  but within `bigint` range (~9.2 × 10^18). No overflow risk at realistic
  scale.
- The append-only trigger means bug-fix corrections require the same
  two-drop, correct, re-create pattern as `ocr_audit_log` (see M-002 §5).
