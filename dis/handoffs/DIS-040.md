## Summary

Integration test at `dis/tests/integration/approve-race.test.ts` drives two
concurrent `orchestrator.approve()` calls on the same extraction at the same
`expectedVersion`. DIS-026's optimistic-lock semantics ensure exactly one
writer succeeds; the other throws `VersionConflictError`.

## Test cases

1. **Concurrent race:** `Promise.allSettled([approveA, approveB])` with the
   same expectedVersion. Asserts exactly one fulfilled, exactly one rejected
   with `VersionConflictError`, the error's `currentVersion === 3`
   (winner's bump), `currentStatus === 'verified'`, and DB row version
   bumped exactly once.
2. **Stale-version sequential:** non-concurrent expression of the same
   invariant — first approve wins, second (with the now-stale
   expectedVersion=2) still raises `VersionConflictError`.

## Notes

The race is legitimate even in single-threaded JS: both coroutines suspend
at `await db.findExtractionById(id)` before either enters
`updateExtractionStatus`. The fake's compare-and-set on `version` resolves
the race deterministically — only the first writer whose expected version
still matches the row's current version succeeds.
