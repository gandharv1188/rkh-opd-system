## 3. Test shape
`describe('DIS-103 — E2E happy path …')` → one `it('extraction reaches promoted state', …)`.
Steps:
1. `POST /ingest` with `application/pdf` + `X-Patient-Id` + `Idempotency-Key` → expect 201, `status='uploaded'`, `version=1`.
2. `orchestrator.process(id)` (no HTTP route for this hop in Wave-5) → expect `ready_for_review`, `version=2`.
3. `GET /extractions/:id` → expect 200, projection matches.
4. `POST /extractions/:id/approve` with `{ expected_version: 2, actor: 'nurse-1' }` → expect 200, `status='verified'`, `version=3`.
5. Terminal hop `verified → promoted` computed via `transition('verified', {kind: 'promoted'})` + `db.updateExtractionStatus(id, 3, 'promoted')` → expect `version=4`.
6. Final `GET /extractions/:id` → expect `status='promoted'`, `version=4`.
7. Invariant: the sequence of `updateExtractionStatus` newStatus values is `['ready_for_review', 'verified', 'promoted']`.

## 4. State-machine event sequence used
Per `dis/src/core/state-machine.ts`:
- `routed_native` (implicit in `orchestrator.process()` native-text branch) → `uploaded → structuring`
- `structured` → `structuring → ready_for_review`
- `nurse_approve` (via `orchestrator.approve()`, called by the approve route) → `ready_for_review → verified`
- `promoted` → `verified → promoted`

There is no `doctor_approve` event in the state machine; the handoff brief's mention of it is obsolete. Only `nurse_approve` exists, which moves `ready_for_review → verified`. The final hop `verified → promoted` uses the `{kind: 'promoted'}` event.

## 5. Why the final hop is driven through the state machine, not HTTP
There is no HTTP route for the `promoted` event in Wave-5 (TDD §13 / DIS-037 wires the real promotion service later). The existing DIS-034 integration test (`tests/integration/state-machine-happy-path.test.ts`) uses the exact same approach — `transition()` to compute the target state plus a direct `db.updateExtractionStatus(...)` write to persist it. Reusing that pattern keeps CS-1 assertable end-to-end (`promoted` is only reachable from `verified` or `auto_approved`) without adding speculative routes.
