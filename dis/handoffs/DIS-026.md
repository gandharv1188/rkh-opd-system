## 3. Decisions taken during implementation

### D-1: Monotonic origin = 1, not 0

**Context:** Need an origin for new rows. Spec says "monotonic integer increments".
**Options considered:** start at 0, start at 1.
**Decision:** `v ≥ 1` is required; `bumpVersion(0)` throws.
**Reason:** DB-side convention across Supabase/Postgres rows seen in sibling epics is `version INT NOT NULL DEFAULT 1`. Rejecting 0 surfaces accidentally-uninitialised rows loudly instead of silently bumping them to 1 and masking the bug.
**Revisit if:** DIS-021a schema lands with a different DEFAULT.

### D-2: `VersionConflictError` lives in `errors.ts`, re-exported from `version-lock.ts`

**Context:** VERIFY-4 accepts the symbol in either file. Multiple tickets in Wave 2a/2b touch `errors.ts`.
**Options considered:** inline class in `version-lock.ts` only; shared `errors.ts`.
**Decision:** shared `errors.ts` as the single source of truth; re-export from the helper.
**Reason:** DIS-029 (error-envelope builder) greps for `VersionConflictError` across core; centralising avoids stale duplicates. DIS-033 follows the same pattern for `NativePdfUnavailableError`.
**Revisit if:** `errors.ts` grows to > ~10 classes — then split by subdomain.

### D-3: Reject non-integer + NaN in `bumpVersion`

**Context:** `Number.isInteger(NaN) === false` and `NaN < 1 === false`, so without the integer check a NaN or fractional version would silently increment.
**Decision:** `!Number.isInteger(v) || v < 1` covers both cases in a single guard.
**Reason:** Data-integrity guardrail at the pure-utility layer; cheaper than arguing about it in adapters.

## 10. Non-obvious gotchas

- `Number.isInteger(Number.MAX_SAFE_INTEGER)` is `true`, so the order of the two `RangeError` guards matters: the `< 1` / non-integer check must come before the `MAX_SAFE` check, otherwise a fractional `2^53` would take the wrong branch.
