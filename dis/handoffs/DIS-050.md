## 1. What was built

- `dis/src/adapters/ocr/datalab-chandra.ts` — `DatalabChandraAdapter implements OcrPort`. POSTs multipart to `https://www.datalab.to/api/v1/convert` with `X-Api-Key` from `SecretsPort.get('DATALAB_API_KEY')`. If the submit response is not already `complete`, polls `request_check_url` with exponential backoff (1s, 2s, 4s, 8s, capped at 10s) until `status === 'complete'`, `status === 'failed'`, or total budget (default 120 s) exhausts. Typed errors: `OcrProviderError`, `OcrProviderTimeoutError`, both stamped with `provider: 'datalab'`.
- `dis/tests/unit/adapters/datalab-chandra.test.ts` — Vitest suite (7 `it(...)` declarations) covering: synchronous-complete submit, submit→poll→complete with observed backoff schedule, submit 500 → `OcrProviderError`, 120 s budget exceeded → `OcrProviderTimeoutError`, `status: 'failed'` → `OcrProviderError`, API key flow from `SecretsPort`, and CS-2 byte-identical `rawResponse`.
- `dis/tests/fixtures/datalab/convert-response.json` — canned provider response (markdown + blocks + `page_count: 2` + `version`) used by fake fetch.

## 3. Decisions taken during implementation

### D-1: Inject `fetch`, `sleep`, `now` rather than mocking globals

**Context:** Tests need to observe backoff waits without actually waiting and must drive a virtual clock for the 120 s timeout test.
**Decision:** Constructor options accept `fetchImpl`, `sleep`, `now`, `maxTotalWaitMs`. Defaults match the production contract (`fetch`, real `setTimeout`, `Date.now`, 120 000 ms).
**Reason:** Keeps the adapter hermetic and deterministic in unit tests. Matches the injection pattern already used in DIS-022 and the coding_standards §Seams guidance.

### D-2: Budget check _before_ sleeping, not after

**Context:** Test asserts a `>120 s` scenario never actually resolves.
**Decision:** Compute `wait` for the next attempt, then compare `totalWaited + wait` against the budget. Throw `OcrProviderTimeoutError` if it would overshoot.
**Reason:** Fail-closed (CS-2 spirit): better to abandon one attempt early than to wait past the declared budget and then check.

### D-3: `rawResponse` is the _final_ provider body, not a merge of submit + poll responses

**Context:** The port contract says `rawResponse` is for audit/reprocessing; callers must be able to replay extraction from it alone.
**Decision:** Store the final `complete` body. Submit-only intermediate responses (202/`submitted`) are not persisted — they hold no OCR content.
**Reason:** Matches CS-2 "reproducible from the raw" intent — the final body contains markdown, blocks, and `page_count`.
**Revisit if:** auditors later need the full submit→poll transcript (then move to a structured `{submit, polls[], final}` envelope).

### D-4: Block type coercion tolerant, not strict

**Context:** Datalab's block taxonomy may add new types over time; our `BlockType` union is closed.
**Decision:** Unknown block types fall through to `'text'` in the `blocks[]` mapping. `rawResponse` still carries the original `block_type` verbatim, so no information is lost.
**Reason:** Forward-compatible — a new Datalab block type won't break ingestion. CS-2 preservation isn't affected because the raw body is untouched.
