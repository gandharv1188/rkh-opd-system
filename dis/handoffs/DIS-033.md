## 1. What was built

- `dis/src/core/native-pdf.ts` — `extractNativeText(bytes: Uint8Array) → Promise<{ pages: [{ page, text }] }>`. Uses lazy dynamic import of `pdfjs-dist/legacy/build/pdf.mjs` so the pdfjs runtime does not load for callers that don't use it. Calls `getDocument` then iterates pages, concatenating `TextContent` items. Throws `NativePdfUnavailableError` when `(totalChars / pageCount) < 5` — heuristic for "image-only PDF / no text layer".
- `dis/src/core/errors.ts` (extended) — added `NativePdfUnavailableError(message, pageCount, totalChars)`. `VersionConflictError` pre-existed from DIS-026 on the same branch.
- `dis/tests/unit/native-pdf.test.ts` — 2 tests: happy path (extract "hello from DIS" from the fixture) + error path (empty content stream → `NativePdfUnavailableError`).
- `dis/tests/fixtures/native_text.pdf` — 595-byte hand-crafted PDF 1.4 with a single Helvetica Tj operator rendering "hello from DIS". Also regenerated inside `beforeAll` if missing, so the test is self-healing on a fresh checkout.

## 3. Decisions taken during implementation

### D-1: Fixture generator + pre-committed fixture

**Context:** Ticket lists `dis/tests/fixtures/native_text.pdf` in `files_allowed`, so the file exists in the tree. But I also need the test to regenerate it if ever deleted/corrupted (reviewer running on a different platform).
**Options considered:** (a) commit the fixture only; (b) generate at test time only; (c) both.
**Decision:** Both — pre-commit the 595-byte PDF AND regenerate-if-missing in `beforeAll`.
**Reason:** Pre-commit satisfies a literal `files_allowed` reading; regen guards against fixture rot or git's LF/CRLF text-mode mangling of the PDF's binary header on Windows clients that mis-classify .pdf (we already saw bytes survive under the current `.gitattributes` defaults; the fallback is cheap insurance).
**Revisit if:** binary attributes are hardened — the regen step can be dropped.

### D-2: Lazy `await import('pdfjs-dist/…')`

**Context:** Brief explicitly calls for lazy import.
**Decision:** `await import(...)` inside `extractNativeText`, not at module top-level.
**Reason:** pdfjs-dist pulls ~1MB of worker/font scaffolding; eager loading would slow every vitest run that doesn't touch this module (state-machine, audit-log, etc.).
**Revisit if:** benchmarks show the per-call import hurts p99 — then memoise the import.

### D-3: Uint8Array copy before handing to pdfjs

**Context:** pdfjs `getDocument({ data })` can internally detach/transfer the buffer, leaving the caller with an empty Uint8Array.
**Decision:** Allocate a fresh `Uint8Array(bytes.byteLength).set(bytes)`.
**Reason:** Callers (file-router) will keep `bytes` around for hashing and OCR fallback if native fails; we must not corrupt their view.
**Revisit if:** we can prove pdfjs no longer transfers — drop the copy.

### D-4: `disableFontFace: true, isEvalSupported: false`

**Reason:** Node environment has no DOM; disabling font-face suppresses the "standardFontDataUrl" warning noise (still prints once but harmless) and hardens against eval-based CMAP loading paths. Matches the pdfjs "Node.js usage" recipe.

### D-5: Heuristic threshold `totalChars / pageCount < 5`

**Context:** Spec text says "< 5". Implemented literally, guarding against `numPages === 0` by treating zero pages as unavailable.
**Reason:** Matches §9.2 wording; keeps the tuning knob at a single constant (`MIN_CHARS_PER_PAGE`).
**Revisit if:** real-world scanned PDFs produce false negatives (e.g., 4-char page numbers from OCR-lite embedded text) — then raise to 10 after measurement.

## 4. What was deliberately NOT done

## 10. Non-obvious gotchas

- The pdfjs runtime prints a one-off `UnknownErrorException: Ensure that the standardFontDataUrl API parameter is provided.` warning to stdout during `getTextContent`. It is non-fatal and does not affect text extraction — do NOT treat it as an error.
- On Windows, `git config core.autocrlf=true` can corrupt the PDF fixture on checkout; the `beforeAll` regenerator guards against this.
- The test file imports the source as `../../src/core/native-pdf.js` (note `.js`) even though the source is `.ts`. This is required by the project's `tsconfig.json` (`moduleResolution: NodeNext`).
