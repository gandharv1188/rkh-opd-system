## 1. What ships

A Hono middleware `killSwitch(deps)` that short-circuits every write
request (POST/PUT/PATCH/DELETE) with HTTP 503 + `Retry-After` when the
kill-switch is enabled. Read requests (GET/HEAD/OPTIONS) pass through
unchanged — clinicians retain read-only access to the record even during
an ingestion freeze.

Response body matches the canonical envelope from
`dis/src/http/middleware/error-envelope.ts`:

```json
{
  "error": {
    "code": "KILL_SWITCH_ACTIVE",
    "message": "Document ingestion is temporarily disabled. Use the legacy flow.",
    "retryable": true,
    "correlation_id": "<uuid from upstream middleware>"
  }
}
```

## 3. Decisions

### 3.1 Fixture strategy — DI predicate, with DB path + env fallback documented

Three activation paths, in precedence order:

1. **`isEnabled: () => boolean | Promise<boolean>`** — explicit predicate.
   This is the test path and, pragmatically, also the recommended
   production path until `dis_kill_switch` exists (§3.2). Accepts
   sync or async.
2. **`db: DatabasePort`** — when provided, the middleware issues
   `SELECT enabled FROM dis_kill_switch LIMIT 1` on every write.
   Single-row config pattern per the brief. The table **does not yet
   exist** in migrations M-001..M-008 (see §3.2), so this path is
   currently exercised only via an injected fake `DatabasePort` in the
   test suite.
3. **No deps** → default off. Chosen over "default on" so a fresh dev
   boot or a misconfigured wiring cannot inadvertently brick writes.

An env-var toggle (`process.env.DIS_KILL_SWITCH_ENABLED === 'true'`) is
**not** implemented in middleware code — it belongs one level up in the
composition root, which can pass `isEnabled: () => process.env.DIS_KILL_SWITCH_ENABLED === 'true'`
in one line. Keeping env lookup out of middleware keeps it pure and
test-friendly (no `vi.stubEnv` gymnastics).

### 3.2 `dis_kill_switch` table — not yet in migrations

A grep of `dis/migrations/*.sql` (M-001..M-008) returns no hits for
`dis_kill_switch`. The brief flagged this as expected ("If the exact
table/column differs … use what actually exists and note it"). The
middleware is coded against the spec'd schema (`dis_kill_switch.enabled`,
single-row) so no code change is needed once the migration lands.

**Follow-up DIS-100-followup-a:** add migration M-009 with:

```sql
CREATE TABLE dis_kill_switch (
  id smallint PRIMARY KEY DEFAULT 1 CHECK (id = 1),
  enabled boolean NOT NULL DEFAULT false,
  reason text,
  updated_at timestamptz NOT NULL DEFAULT now(),
  updated_by text
);
INSERT INTO dis_kill_switch (id, enabled) VALUES (1, false);
```

### 3.3 Retry-After default = 300s

The default is 5 minutes — long enough to avoid tight-loop retries from
clients, short enough that a transient freeze doesn't back up a whole
OPD session. Configurable via `retryAfterSeconds` for tests and for
ops scenarios where a longer pause is desired.

## 8. Clinical safety (CS-9)

Kill-switch is the documented emergency stop for the ingestion
pipeline. It must:

- block every state-changing request (✓ — WRITE_METHODS set),
- leave read paths open so clinicians can still retrieve prior records
  (✓ — GET/HEAD/OPTIONS untouched),
- emit a retryable signal so client-side auto-retry does not treat it
  as permanent failure (✓ — `retryable: true` + `Retry-After`),
- carry a stable machine code for dashboards/alerts (✓ —
  `KILL_SWITCH_ACTIVE`).

## 11. Risks

- Default-off is a deliberate footgun trade: if ops believes the switch
  is wired but the composition root forgets to pass `isEnabled`/`db`,
  the switch silently does nothing. Mitigation: DIS-100-followup-b
  asserts one of the two is always set when `killSwitch()` is mounted,
  via a run-time check that logs a WARN if both are undefined.
