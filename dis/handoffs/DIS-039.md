## 1. What was built

`dis/tests/integration/idempotency-store.test.ts` composes the DIS-025
`createIdempotencyStore(db)` factory with an extension of the DIS-012
`FakeDatabaseAdapter` (`IdempotencyFakeDb`) that simulates the
`idempotency_keys` table in-memory. The suite covers all three
`recordAndResolve()` outcomes plus key isolation:

- `action=new` — unseen key inserts one row; 1 transaction + 2 queries
  (SELECT + INSERT).
- `action=replay` — same key + same payload hash returns the existing
  record, re-insert does NOT happen (`db.table.size` stays at 1).
- `action=collision` — same key + different hash returns the ORIGINAL
  record (not the incoming hash), stored row unchanged.
- Key isolation — two distinct keys both resolve to `new`.

Unlike the DIS-025 unit tests, these treat store + fake-DB as a composed
unit and assert observable ordering (tx count, insert count, table
identity) rather than the store's SQL shape.

## 3. Decisions

### D-1: Local `IdempotencyFakeDb` instead of shared helper

The unit tests (`dis/tests/unit/idempotency-store.test.ts`) define the same
small extension. Duplicating it here keeps the integration suite
independently modifiable — a future addition of another simulated table
shouldn't couple to the unit-test fixture's schema.

### D-2: Assert observable side-effect counts, not SQL shape

The unit tests already cover "parameterised SQL" and "uses transaction".
The integration tests assert what a caller sees — row count after
dedupe, `db.table.size`, insert-call count — so a regression in
semantics would surface even if the SQL wording shifts.
