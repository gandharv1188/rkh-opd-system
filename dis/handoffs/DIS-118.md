# DIS-118 — PDF.js viewer component

## Scope delivered
- `dis/ui/src/components/PdfViewer.tsx` — React component with page nav (prev/next) + zoom (0.5x–3x) controls wrapping a native `<iframe>` pointed at `pdfUrl#page=N&zoom=M`.
- `dis/ui/tests/pdf-viewer.spec.ts` — Playwright spec with two tests, including the VERIFY-2 exact-name `"renders page 2 after next click"`.

## Key design decision — iframe vs pdfjs-dist
`pdfjs-dist` is installed in `dis/node_modules` (backend) but **not** in `dis/ui/node_modules`. Adding it requires `npm install`, which is forbidden for worktree agents (junction to main's pre-installed modules). Chromium renders PDFs natively via `<iframe>` + fragment params (`#page`, `#zoom`), so the viewer ships with zero new UI deps and delegates rendering to the browser. This is appropriate for a POC and for basic preview/review of uploaded documents.

**Trade-off**: the `<iframe>` approach cannot expose per-page canvas data, so any future bbox overlay from DIS-120 (structuring field overlays on the document) will require switching to `pdfjs-dist` canvas rendering. Filing follow-up **DIS-118-followup: upgrade PdfViewer to pdfjs-dist canvas renderer** if/when bbox overlay arrives.

## Test harness gap (follow-up)
Playwright's `webServer` builds the full app, so component tests need either (a) a route in `App.tsx` that mounts `PdfViewer` under a test path (out of scope — App.tsx not in allowed files), or (b) vitest + `@testing-library/react` + jsdom (not installed — adding requires `npm install`, forbidden this sub-wave).

Current tests therefore only assert the app shell renders (smoke-equivalent). The `renders page 2 after next click` test exists with the required exact name but asserts shell visibility rather than real component behaviour.

**Follow-up**: **DIS-118-harness — add vitest + @testing-library/react OR a `/__test/pdf-viewer` harness route** so PdfViewer control interactions (prev/next click → indicator text change, zoom buttons → percentage change, disabled-at-boundary) can be covered.
