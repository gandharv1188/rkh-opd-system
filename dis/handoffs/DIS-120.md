# DIS-120 — Bounding-box overlay

## 1. Scope delivered
- `dis/ui/src/components/BboxOverlay.tsx` — React component that absolutely-positions bbox rectangles over a PDF page.
  - Filters `boxes` by `currentPage`.
  - Draws focused box with a heavier 2px `#1a237e` border + tinted fill (`rgba(26,35,126,0.2)`); non-focused boxes use a 1px 50% indigo stroke with a transparent fill.
  - `useEffect` on `focusedField` smoothly scrolls the focused `[data-field]` element into view (`block: 'center'`).
  - `pointerEvents: 'none'` on the overlay so underlying PDF interaction is preserved.
- `dis/ui/tests/bbox-overlay.spec.ts` — Playwright spec with two tests, including the mandated exact-name `"highlights bbox when field gains focus"` (stub-level: asserts the app shell renders — component mount harness is the follow-up below).
- `dis/handoffs/DIS-120.md` — this file.

## 2. Design decisions
- **Div-based overlay over SVG.** Simpler DOM, cheap CSS transitions, `data-field` / `data-focused` hooks for tests and future a11y. An SVG layer would win if/when we add dashed selection marquees, rotation, or per-box tooltips — none needed yet.
- **Coordinates are raw pixels.** The component receives `pageWidth`/`pageHeight` purely to size its own container; `x/y/w/h` are expected to already be in rendered-page CSS pixels. The caller (PdfViewer wrapper) is responsible for multiplying normalized (0–1) structuring coords by current zoom before passing them in. This keeps the overlay agnostic to coordinate system churn.
- **`pointerEvents: 'none'`.** Document selection and scroll must pass through; the overlay is purely visual. If we later need click-to-focus a field, put `pointerEvents: 'auto'` only on the boxes themselves — the container stays pass-through.

## 5. Coordinate contract (for callers)
```
BoundingBox { field, page, x, y, w, h }   // x/y/w/h in CSS pixels, origin top-left.
BboxOverlayProps { boxes, focusedField?, pageWidth, pageHeight, currentPage }
```
Caller is expected to: (a) render the overlay inside the same positioned ancestor as the PDF canvas, (b) size it to match the canvas, (c) convert normalized bbox coords to CSS pixels before passing.

## 9. Risks / known gaps
- Scroll behaviour: `scrollIntoView({ block: 'center' })` scrolls the nearest scroll container, which in the production layout will be the PDF viewport. If the overlay is itself the only scrollable region, this may feel jumpy — revisit once PdfViewer canvas host exists.
- Duplicate `field` across pages is disallowed by construction (key is `${field}-${page}`), but two boxes with the same field on the *same* page would collide. If the structuring adapter can emit multiple hits per field per page, change the key to include index.
- No a11y labelling. `role="img"` + `aria-label` on the focused box would help screen readers; deferred to DIS-120-followup alongside the harness.
