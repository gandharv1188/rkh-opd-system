## 1. Starting state

The session opened on two threads that had been left dangling. The Supabase dashboard had been showing an IO budget warning for several days, and a previous session had hypothesised that four fixes — preloading the formulary, throttling the 30-second autosave, replacing the Realtime websocket, and adding trigram indexes — would absorb the spike. None of that had been verified against actual database metrics. In parallel, the doctor had been complaining that prescriptions kept arriving with the wrong list of medications: drugs he had not asked for were appearing, drugs he had typed were quietly disappearing, and brand names he wrote in his note were being silently rewritten. That complaint had been acknowledged but never properly investigated. The session began with the goal of closing the IO loop with real data, then turning attention to the medication problem.

## 2. The IO investigation — using Supabase MCP for the first time

The first move was to wire up the Supabase MCP server in read-only mode so we could ask the database itself what was consuming buffers. There was a frustrating loop on environment variable substitution — `${SUPABASE_ACCESS_TOKEN}` was not being expanded by the MCP loader on Windows — which we resolved by hardcoding the access token directly into `.mcp.json` and adding that file to `.gitignore`. With the connection live, we ran `pg_stat_statements` and pulled the top consumers by `shared_blks_hit + shared_blks_read`.

The result reframed the problem. Realtime polling — the websocket we had assumed was a small contributor — was responsible for roughly 97% of all buffer hits. Formulary preloading, which the previous session had treated as the headline issue, came in well under 1%. The four-fix list was re-prioritised on the spot: kill Realtime first (95% impact), disable the 30-second autosave (around 0.3%), trim the formulary preload last (mostly a latency win, not an IO win). Trigram indexes were dropped from the plan altogether after `EXPLAIN (ANALYZE, BUFFERS)` showed the formulary table is small enough that sequential scans are already cheap.

A second insight surfaced while we were considering whether to narrow the formulary `SELECT` columns. Postgres TOAST'd JSONB columns are fetched on demand, so trimming the column list does not reduce heap reads — it only saves CPU and serialisation time. That observation was written into doc 09 so the reasoning would not be re-litigated next time someone looks at the formulary endpoint.

## 3. Shipping the IO fixes

The two highest-impact changes went out the same afternoon. In `web/prescription-pad.html`, the Realtime websocket subscription was replaced with a 60-second `setInterval` poll, and the 30-second autosave timer was commented out (debounced save on blur was kept). We verified the deploy in Chrome via the Chrome DevTools MCP: the page connected without a websocket, the 60-second poll fired on schedule, `autosaveTimer` was `null`, and the console was clean. Commit `1d80756` was pushed and GitHub Pages redeployed.

The doctor then disabled the Realtime publication in the Supabase dashboard manually, toggling all four event types off on every published table. We rechecked `pg_stat_statements` five minutes and forty-four seconds later and saw zero new realtime calls — confirmation that polling had fully stopped, not just throttled. Doc 09 was rewritten with the real numbers replacing the earlier guesses, and doc 10 was added to capture the side-effect analysis (what we lose by giving up Realtime, what the polling fallback covers, and what stays broken — namely the cross-tab "patient marked done" notification).

## 4. Pivot to AI implementation audit

With the IO fire out, attention turned to the prescription quality complaint. A code-reviewer subagent was dispatched against the Edge Function (`supabase/functions/generate-prescription/index.ts`) and the skill files in `radhakishan_system/skill/`. The audit returned three CRITICAL findings. C1: `web/dose-engine.js` exists and is correct, but it is only invoked when the doctor manually clicks "Adjust dose" — the AI's emitted doses are never validated against it. C2: when `get_formulary` returns no match for a drug name, the tool's response string ends with the sentence "Use your clinical training knowledge for dosing." That is the exact failure mode the doctor had been describing. C3: the `overall_status` field on the prescription JSON (SAFE / REVIEW REQUIRED) is whatever string the AI chooses to emit; nothing on the server side computes it from the actual safety flags. Six HIGH-severity findings followed, including a missing weight guard before generation, no validation of the Devanagari output, no preterm corrected-age handling, no edit history table, and silent fallback when the formulary tool fails. These were captured in doc 11.

## 5. Customer feedback reframes the priority

The doctor was clear: the number-one pain is medication selection, not dose math. To verify that under controlled conditions, we used Supabase MCP to pull eight recent prescriptions and audited each against the underlying `visits.raw_dictation`. The complaint held up. Five of eight visits had silent additions; three had drops; three had brand-to-generic substitutions that did not match what the doctor had typed; two had internal inconsistencies between the calculated dose and the displayed dose. The clearest case was RX-260427-018 (patient HETANSHI), where the doctor had asked for two medicines and the AI returned six — and the four extras matched the standard asthma/rhinitis protocol's first-line list exactly. That match was not coincidence; it was structural. Sprint 1's headline goal pivoted from dose validation to medication-list integrity.

## 6. Research before designing

Before drafting the fix, two research subagents went out in parallel. The architecture agent (output: doc 13) surveyed "LLM proposes / deterministic engine validates" patterns and surfaced Goodell et al. 2025 in *npj Digital Medicine*: deterministic dose-calculation tools beat raw LLM output 95% to 36% on accuracy, and beat code-interpreter approaches 95% to 46%. The verdict on our previous three-layer dose plan was that Layers 1 and 2 were directionally correct, but Layer 3 — silently overriding the AI's dose if it differed by more than 20% — was unsafe and NABH-non-defensible for narrow-therapeutic-index drugs. The recommended replacement was a six-stage pipeline with explicit pre- and post-validation, no silent overrides, and human acknowledgement on every safety divergence.

The prompting agent (output: doc 14) worked the medication-omission problem from the LLM side. Root causes: there is no enumeration step in the output schema (no `requested_medicines[]` echo of what the doctor typed), `tool_choice: auto` allows serial tool calls that consume the context window, long tool loops are subject to mid-stream compaction, and `max_tokens` may be truncating completions before the medicines array closes. Recommendations: add the echo field with a deterministic post-call diff, drop temperature to 0 (it had been 0.2), and remove `overall_status` from the LLM's responsibilities entirely — compute it server-side from the structured flags.

## 7. Standard Rx flow trace

The doctor pointed out something the audit had missed: the Standard Rx button on the Prescription Pad was originally designed as a *user-triggered* context — the doctor would pick a protocol and the AI would use it as a reference. A trace subagent was sent through the full path (output: doc 12). The findings were uncomfortable. The Std Rx chip is a binary toggle with no protocol selector, and it defaults to ON. The frontend pre-fetches all 503 protocols into `stdRxCache` but never reads from the cache. The POST body sent to the Edge Function contains no `std_rx_context` field; the function destructures one but never uses it (dead code). What actually drives protocol injection is the AI's autonomous call to `get_standard_rx`, instructed by `core_prompt.md` to "ALWAYS call this when a diagnosis is provided." The Std Rx button itself only appends "USE STANDARD PRESCRIPTION include ALL first-line drugs" to the user prompt. That is the structural cause of the contamination we saw in the HETANSHI case. The fix path written into doc 16 is: button defaults OFF, the frontend pushes the doctor's selected protocol into the prompt as a `<doctor_selected_protocol>` block, and the AI's auto-fetch tool is gated to only fire when the button is ON.

## 8. Decision-by-decision walkthrough

With the diagnosis clear, we worked through 37 design decisions using `AskUserQuestion`, captured in full in doc 15. They group into seven themes.

**Medication handling (decisions 1–3, 32).** The AI must never silently drop a drug the doctor named — drops are blocking errors. Additions are only allowed when the Std Rx button is explicitly ON, and even then they appear as a separately-styled "Suggested additions" block the doctor must accept. Brand-to-generic substitutions follow NABH format: GENERIC NAME (Brand). The Std Rx toggle resets to OFF on every patient change so it cannot leak across visits.

**Dose math (decisions 4–5).** When the doctor writes a dose explicitly, the doctor's dose wins, but if the deterministic engine flags it as outside the safe band, a banner appears and the doctor must acknowledge. For implicit dosing, the AI calls a new batched `compute_doses` tool rather than doing mental math.

**Diagnosis matching (decision 6).** Two layers: a curated menu in the prompt for the top diagnoses, plus a `pg_trgm` fuzzy match in the existing tool. No separate AI classification call.

**Severity (decisions 7, 8, 31).** Three FHIR-aligned tiers (mild/moderate/severe). The displayed severity is `max(server_computed, ai_emitted)`. The banner-and-acknowledge UX pattern from dose math is reused for every safety gate so the doctor only learns one pattern.

**Edge cases (9–14).** Formulary misses surface fuzzy suggestions instead of falling back to "use your training." Allergy clashes propose an alternative drug and require an override checkbox. Weight is mandatory at the nurse station before generation is allowed. Preterm corrected age is computed in the front end so the AI never has to. The bilingual library is extended in place. Edit history is append-only.

**Privacy and security (15, 16, 17, 22, 23, 25, 28, 29, 30).** UHID is masked in logs. Prompt caching is deferred (cost analysis below). The asymmetric Haiku verifier is built but feature-flagged off. Supabase Auth lands this sprint; full RLS roles are next sprint. OCR pre-processing is deferred until the patient is saved. The legacy `postMessage` bridge is deleted. Voice transcription strips patient names before sending. ABHA numbers do not go on the QR code.

**Operations (18–21, 26, 27, 33–37).** A `prescription_audit` table is added for full diff history. Receipt sequence numbers become atomic. The destructive sample-data script gets a project-ref guard. Schema drift is fixed by taking a `pg_dump` baseline. The vaccine "Other" field is whitelisted. Allergy capture is mandatory at registration. Loop monitoring is added to the Edge Function. Low- and medium-severity hygiene items are deferred to the final sprint. The model is upgraded to Sonnet 4.6, temperature is set to 0, and the formulary error message is rewritten to be structured instead of advisory.

## 9. What was deferred and why

A few items got close to inclusion and then stepped back. Prompt caching would save roughly $1–2 a month at clinic volume, and because the prompt is going to mutate repeatedly across Sprint 1, every change would invalidate the cache; revisit once the prompt stabilises. The asymmetric Haiku verifier is in the codebase but flagged off — turning it on costs latency and tokens, and we want to make that decision against real audit data, not speculation. The original "block sign-off entirely on fallback" plan was replaced with the banner-plus-checkbox pattern; the doctor needs a way through when something legitimately is not in the formulary, just not a silent way through. Multi-model constellations and a full Program-of-Thoughts sandbox were both ruled out as cost-prohibitive at this stage.
