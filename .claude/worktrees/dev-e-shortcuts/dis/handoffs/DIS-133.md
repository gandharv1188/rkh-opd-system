# DIS-133 — useShortcuts hook (Enter/Esc/N/P)

## 1. What was built
A keyboard-shortcut layer for the Verify flow, exposed as two artefacts:
- `matchShortcut(key, opts)` — pure helper mapping a `KeyboardEvent`-like shape to one of `'approve' | 'cancel' | 'next' | 'prev' | null`. Side-effect free, trivially testable without a DOM.
- `useShortcuts(handlers)` — React hook installing a single `window` `keydown` listener, delegating to `matchShortcut`, `preventDefault`ing on match, and dispatching the matching callback (`onApprove` / `onCancel` / `onNextField` / `onPrevField`). Supports a `disabled` flag for modal / dialog contexts.

Bindings: `Enter` = approve · `Escape` = cancel · `N`/`n` = next field · `P`/`p` = prev field.

## 3. Decisions taken during implementation
- **Pure helper exported separately** — keeps unit tests deterministic and runner-agnostic; no jsdom dependency.
- **`disabled` flag on the handlers object** rather than a second hook argument — fewer call-sites need to restructure state.
- **Shift allowed** — otherwise capital `N`/`P` (when Shift is held) would be suppressed. Ctrl/Meta/Alt remain blocked so we never shadow browser/OS shortcuts (Ctrl+N, etc.).
- **Tag-name guard via `opts.targetTag`** — lets the pure helper be tested without synthesising a DOM target. Hook forwards `(e.target as HTMLElement | null)?.tagName`.
- **Test harness = Playwright `test`** — mirrors `tests/flags.spec.ts` precedent from DIS-131; vitest is not in `devDependencies`. The required `test('N advances focus to next field', …)` name is preserved verbatim.

## 10. Non-obvious gotchas
- `matchShortcut` is case-sensitive on letter keys because browsers emit `'n'` when Shift isn't held and `'N'` when it is — both map to `'next'` by design. Don't "normalise" to lowercase; that would break the explicit Shift-allowed semantics.
- The hook's `useEffect` depends on the whole `handlers` object. Callers that want to avoid re-binding `keydown` each render should memoise their handlers (`useMemo` / `useCallback`) or hoist them.
- `preventDefault()` fires on every matched shortcut, including `Enter`. This is correct for the Verify flow (Enter must not also submit a stray form) but callers embedding this hook in form-heavy pages should set `disabled: true` when that behaviour is unwanted.
- Input suppression uses `tagName` only. Contenteditable `<div>`s and custom rich-text components will still fire shortcuts — pass `disabled: true` when focus enters such a widget.
