# 15 — Decisions Record (2026-04-28): AI Behaviour, Safety, Security, and UX

_Last updated: 2026-04-28_

This document captures explicit decisions made during a working session on 2026-04-28 reviewing docs 09 through 14. Each entry is a doctor's choice (the product owner's call), not an AI recommendation. The session covered AI behaviour around medication omission and addition, dose engine integration, the Standard Rx button, allergy handling, fallback mode during outages, security and authentication, schema drift, and audit/instrumentation. Implementation steps, sprints, and file-level changes live in doc 16 (the consolidated plan). The purpose of this file is so the team can refer back later and answer "why did we decide X?".

---

## 1. Medication omission

**Decision:** AI never silently drops a drug the doctor wrote. It must include the drug as a stub with a reason. If the AI is completely silent about a drug, the server auto-retries once.

**Rationale:** Silent omission is the worst clinical failure mode — the doctor believes the drug is on the prescription when it is not. A stub with reason makes the omission visible. The existing prescription pad banners already surface stub reasons, so no new UI is needed.

**Source:** doc 14 Section A; customer-reported issue.

---

## 2. Medication addition

**Decision:** With the Standard Rx button OFF (default), the AI prescribes ONLY drugs the doctor wrote. With Standard Rx ON, the AI may add protocol drugs marked "AI suggestion" with a reason; the doctor approves them at sign-off.

**Rationale:** Doctor authority is the default. Protocol-driven prescribing is opt-in per patient and clearly attributed so the doctor never confuses their own choices with AI additions.

**Source:** doc 12 Standard Rx flow gap.

---

## 3. Brand name handling

**Decision:** Output format is `GENERIC NAME (Brand)`. NABH requires generic. If a brand cannot be confidently mapped via the formulary's `brand_names` array, print generic only — no AI guessing from training data.

**Rationale:** A real audit case showed the AI mapping "Vitafol" to Folic Acid from training memory, which was wrong. Confident mapping must come from the formulary, not the model's prior.

**Source:** doc 14; real audit case (Vitafol → Folic Acid misattribution).

---

## 4. Doctor-specified dose

**Decision:** Print exactly what the doctor wrote. The dose engine never silently overrides. If the engine flags the dose as unsafe (above max single or daily dose), a prominent red banner appears with an acknowledge checkbox required before sign.

**Rationale:** Doctor authority is preserved while safety bounds are still surfaced. Silent overrides erode trust; explicit acknowledgement keeps the doctor in control and creates an audit trail.

**Source:** doc 11 CRITICAL-1; doc 14.

---

## 5. Implicit dose (doctor didn't specify)

**Decision:** AI picks a dosing band from the formulary, then calls a `compute_doses` tool that runs the dose engine. The engine returns canonical numbers for ALL drugs in ONE batched call (not per-drug). The AI copies engine output into the prescription.

**Rationale:** Goodell 2025 (cited in doc 13) shows tool-computed arithmetic at 95% accuracy versus 36% for AI mental math. Batching avoids tool-loop bloat. The dose engine is already the source of truth in this codebase.

**Source:** doc 13 architecture (Goodell 2025: 95% vs 36%).

---

## 6. Diagnosis-to-protocol matching

**Decision:** Two layers: (a) send the full menu of 503 (icd10, diagnosis_name) pairs to the AI in the cached system prompt; (b) replace the SQL ILIKE in `get_standard_rx` with pg_trgm similarity matching to handle abbreviations like "AOM". No separate AI call.

**Rationale:** ILIKE misses common clinical abbreviations entirely, which is why protocols rarely match today. Trigram similarity is a single index change, not an architectural one. Giving the AI the full menu lets it pick correctly the first time.

**Source:** doc 12; new finding (no matching exists today).

---

## 7. Fallback mode (Anthropic outage)

**Decision:** Prescription generates with a red banner. An acknowledge checkbox sits next to the Sign button. Sign is disabled until ticked.

**Rationale:** During outage the system must still function, but the doctor must consciously accept the degraded state before signing.

**Source:** doc 11 MEDIUM-1.

---

## 8. overall_status (safety verdict)

**Decision:** Three-tier severity (high / moderate / low) aligned with FHIR DetectedIssue.severity. Both server AND AI compute their own severity; the final verdict is `max(server, AI)`. The AI's textual concerns are kept as a separate `ai_safety_notes` field.

**Rationale:** Server-only checks miss the AI's clinical eye on unusual cases. Safer-of-the-two is the standard pattern in clinical decision support. FHIR alignment makes downstream ABDM work simpler.

**Source:** doc 11 CRITICAL-3; doc 14 Section C.

---

## 9. Formulary not-found (drug missing from formulary)

**Decision:** Print as a stub: `WIKORYL — DOSE TO BE VERIFIED MANUALLY`. Show the reason. Server runs fuzzy search on `brand_names` and suggests similar drugs inline. Force `overall_status = high`. The doctor fills the dose manually or clicks a suggestion.

**Rationale:** A missing drug must never be silently omitted nor silently dosed. The stub keeps it visible, the suggestion accelerates correction, the high severity blocks unconsidered sign-off.

**Source:** doc 11 CRITICAL-2; doc 14 Section B.

---

## 10. Allergy clash

**Decision:** The doctor's drug stays printed (authority preserved). The AI adds a prominent warning plus a suggested alternative. The Sign button is disabled until the doctor either swaps to the alternative OR ticks an override checkbox.

**Rationale:** Hard-blocking the prescription erodes authority; silent printing erodes safety. The middle path is to print, warn loudly, and require explicit acknowledgement.

**Source:** Clarification during 2026-04-28 session.

---

## 11. Missing weight

**Decision:** Primary fix: the nurse cannot save a visit without weight (mandatory field at the nurse station UI). Backup: if weight is still missing when Generate is clicked, prompt the doctor with an inline input that saves to the visit and continues.

**Rationale:** Weight-based dosing requires weight. Catching the gap upstream at the nurse station prevents most cases; the doctor-side prompt is a safety net.

**Source:** doc 11 HIGH-1.

---

## 12. Preterm corrected age

**Decision:** The front-end computes `corrected_age_days` and `chronological_age_days` when the neonatal chip is active (GA<37wk OR DOB<90d OR BW<2.5kg). Both values are sent as structured fields to the AI. The AI does no arithmetic.

**Rationale:** Age arithmetic for preterms is a documented AI failure mode. Deterministic computation in JavaScript is trivial and removes a class of error.

**Source:** doc 11 HIGH-3.

---

## 13. Hindi text (row3_hi)

**Decision:** Boilerplate (doses, frequencies, durations, routes) is composed deterministically from existing dose-engine.js Hindi maps. Unusual clinical instructions are translated by the AI. The server validates Devanagari Unicode (U+0900–U+097F) presence; if missing, force `overall_status = high` with a "Hindi invalid" flag. The library is extended over the next sprint with about 30 clinical phrases.

**Rationale:** Hindi rendering errors are frequent and patient-facing. Deterministic composition for the 90% case is reliable; AI handles the long tail; server validation catches both paths.

**Source:** doc 11 HIGH-2.

---

## 14. Prescription edit history

**Decision:** Append-only `prescriptions_history` table. Every save (initial plus edits) inserts a row with `prescription_id`, `version_number`, `generated_json`, `edited_by`, `edited_at`, `edit_reason`. The original signed version is never overwritten. The print page shows a "Version N of M" tag.

**Rationale:** NABH and medico-legal requirements mandate that the original signed prescription be preserved. Append-only is the simplest pattern that satisfies both.

**Source:** doc 11 HIGH-3 (NABH/medico-legal).

---

## 15. PHI in Edge Function logs

**Decision:** Mask UHID in all `console.log` calls: replace `patient_id` with first 4 + last 4 characters (e.g., `RKH-****0355`). Apply across `executeGetPreviousRx`, `executeGetLabHistory`, and the main `executeTool` log.

**Rationale:** Edge Function logs are retained by Supabase and are visible to anyone with project access. Masking is a low-cost privacy improvement aligned with DPDP Act expectations.

**Source:** doc 11 MEDIUM-4.

---

## 16. Prompt caching

**Decision:** Deferred. Revisit after the prompt stabilises (about 1–2 weeks post Phase 1).

**Rationale:** At 20 prescriptions/day with clusters within 5 minutes, the savings are about $1–2/month. The prompt is changing during this sprint, which would invalidate the cache on every change. The optimisation is real but not yet worth the engineering attention.

**Source:** doc 14 Section E; verified Anthropic pricing 2026-04-28.

---

## 17. Asymmetric verifier (Haiku second pass)

**Decision:** Build into the Edge Function as a feature-flagged capability (default OFF). Enable on a small percentage of cases first to compare against actuals. The decision to roll out is data-driven later.

**Rationale:** A second-model verifier adds latency and cost; whether it catches enough real errors to justify those costs is empirical. Build the capability, instrument it, decide later.

**Source:** doc 13 Section 4.

---

## 18. Instrumentation table

**Decision:** New `prescription_audit` table with: `prescription_id` (FK), `stop_reason`, `input_tokens`, `output_tokens`, `tools_called[]`, `requested_meds[]`, `omitted_meds[]`, `added_meds[]`, `verifier_flags` (jsonb), `meta_mode` (tool-use vs fallback), `created_at`. One row per generation attempt including retries.

**Rationale:** Without instrumentation, every other decision in this document is unmeasurable. This is the foundation for evidence-based iteration on the AI behaviour.

**Source:** doc 14 Section E.

---

## 19. Receipt collision (per-tab counter bug)

**Decision:** Postgres sequence-based atomic claim. Either daily sequence reset via `pg_cron`, OR atomic `SELECT max(receipt_no)+1 FROM visits WHERE visit_date = today FOR UPDATE`. Frontend reserves the number via a small RPC. Collision-free across reception PCs.

**Rationale:** The current per-tab counter collides whenever two reception PCs register a patient in the same minute. The fix must live in the database, not the browser.

**Source:** doc 11 H8.

---

## 20. create_sample_data.js project guard

**Decision:** Add a 3-line check at the top: `if (process.env.SUPABASE_PROJECT_REF !== 'dev-only-ref') throw 'Refusing to run on non-dev project';`.

**Rationale:** The script auto-scrubs and reseeds. Running it against the live project would wipe real patient data. A trivial guard prevents an existential accident.

**Source:** doc 11 H7.

---

## 21. Schema drift

**Decision:** `pg_dump --schema-only` from live, commit it as a new baseline. Adopt the standard `supabase/migrations/` folder going forward. Every schema change becomes a numbered migration committed before being applied to live.

**Rationale:** The repo schema and the live schema have drifted. Re-baselining once and adopting the standard migration workflow stops the drift permanently.

**Source:** doc 11 H1+H2.

---

## 22. Authentication and security

**Decision:** Phase A (this sprint): add Supabase Auth email/password login on every page. Anyone without a valid session is redirected to login. The anon key stays only for the login flow. RLS remains at `anon_full_access` for now. Phase B (next sprint): refine RLS into per-role policies (doctor / nurse / reception). Audit log of who-did-what alongside `prescription_audit`.

**Rationale:** Supabase warning emails flag the current state. Splitting into two phases lets us close the unauthenticated-access hole immediately while taking the time to design role policies properly.

**Source:** Supabase warning emails; audit doc 11 M2.

---

## 23. Hardcoded credentials cleanup

**Decision:** Bundle with the auth work. While touching every HTML page to add the login redirect, also extract `SUPABASE_URL` and `ANON_KEY` into `web/config.js`. Single pass.

**Rationale:** Both changes touch the same files. Doing them together avoids two rounds of edit-test-deploy.

**Source:** doc 11 M1.

---

## 24. External QR dependency

**Decision:** Vendor a small JS QR library locally (about 20 KB, e.g., qrcode.js). Replace `api.qrserver.com` calls with browser-side generation. Prescriptions generate QR offline.

**Rationale:** The current dependency on a third-party QR service means PHI in the URL is sent to an external host, and prescriptions fail if that host is down. A 20 KB library eliminates both risks.

**Source:** doc 11 M9.

---

## 25. Pre-OCR PHI upload

**Decision:** Hold the image in browser memory only. Upload to the OCR Edge Function only after `patient_id` exists (after registration save). If the user cancels, the image was never sent.

**Rationale:** Currently images can be uploaded before a patient record exists, leaving orphaned PHI in storage. Deferring the upload couples it to a real patient and to the user's commitment to register.

**Source:** doc 11 M6.

---

## 26. Vaccine "Other..." silently dropped

**Decision:** Build a whitelist of common pediatric extras: Vitamin A, BCG, Typhoid, Influenza, Cholera, Hepatitis A, etc. Adding "Other..." opens a dropdown of these instead of free text. Free text fallback remains for rare cases. All entries save properly to the `vaccinations` table.

**Rationale:** Free text "Other..." entries are currently dropped on save. A whitelist covers the common cases cleanly while still allowing the rare ones.

**Source:** doc 11 H4.

---

## 27. NABH 'asked, none' allergy state

**Decision:** The allergy field becomes mandatory at registration. Reception cannot save a patient without selecting NKDA or ALLERGY (with drug list). There is no "NOT ASKED" state — it is eliminated by forcing an answer.

**Rationale:** NABH requires the allergy question to be asked and recorded. Eliminating the implicit "not asked" state by making the field mandatory is simpler and more reliable than tracking it.

**Source:** doc 11 H6.

---

## 28. postMessage listener

**Decision:** Delete entirely. It is dead code from the Claude.ai artifact era.

**Rationale:** Removing attack surface is better than guarding it. The listener serves no purpose in the standalone web app architecture.

**Source:** doc 11 M4 + M3.

---

## 29. Voice transcription PHI

**Decision:** Strip patient name and UHID from `patient_context`. Keep sex, weight, and chief complaints (which help transcription accuracy without identifying the patient). Edge Function logs no longer contain patient names.

**Rationale:** Transcription quality benefits from clinical context, not from identifiers. Stripping name and UHID removes PHI from a third-party API call and from logs at the same time.

**Source:** doc 11 M5.

---

## 30. ABHA on QR code

**Decision:** The QR contains UHID + name initial + DOB + sex only. ABHA is stripped. ABHA stays in the database linked to UHID, used for ABDM API calls when the patient gives consent (DPDP Act / ABDM consent model alignment). Same identification convenience on next visit (scan UHID, look up patient including ABHA).

**Rationale:** ABHA on a printed QR effectively makes the national health ID public. Keeping it server-side behind consent aligns with DPDP and the ABDM consent model without sacrificing the scan-on-return-visit convenience.

**Source:** doc 11 M7.

---

## 31. High-severity flag sign-off gate

**Decision:** Same flow as fallback mode: banner at top + acknowledge checkbox next to the Sign button + Sign disabled until ticked + audit log records explicit acknowledgement.

**Rationale:** A single consistent UX pattern for every safety gate is easier to learn, easier to audit, and easier to maintain than per-case dialogs.

**Source:** doc 11 H9.

---

## 32. Standard Rx button default state

**Decision:** Resets to OFF for every new patient session. The doctor must consciously decide to enable protocol-driven prescribing per patient. No carryover state.

**Rationale:** Carrying state across patients is how protocol drugs end up on the wrong prescription. A per-patient conscious choice is safer.

**Source:** doc 12; clarification during 2026-04-28 session.

---

## 33. Tool-use loop limits

**Decision:** Keep `MAX_TOOL_LOOPS = 10` and the 120s timeout as hard ceilings. Extend the existing early-stopping heuristic at line 569 (same tool/input twice → end loop). Log every generation that needs ≥ 5 rounds to `prescription_audit` with a warning flag.

**Rationale:** The current ceilings are sound. The early-stop heuristic catches most pathological loops; logging the long ones gives us evidence to tune further.

**Source:** doc 11; observed code patterns.

---

## 34. L1–L13 + M2/M3 + M10/M11 (low/medium hygiene)

**Decision:** Stay in the plan as the **final sprint**. Not deferred to a backlog folder; remain on the roadmap, just at the bottom.

**Rationale:** These are real issues, just lower priority than the safety and security work above. Keeping them on the roadmap (rather than in a backlog) means they actually get done.

**Source:** doc 11 lower-priority sections.

## 35. Model upgrade

**Decision:** Already staged (Sonnet 4 → Sonnet 4.6). Will deploy with the first Edge Function deploy of safety changes.

**Rationale:** Bundling the model upgrade with the safety changes means one deploy, one validation cycle.

**Source:** 2026-04-28 session.

## 36. Temperature setting

**Decision:** Set to 0 (not 0.2) on both the main loop and the fallback.

**Rationale:** With strict tool use, the schema constraint dominates anyway, so the diversity argument for temperature 0.2 is weak. Reproducibility for clinical audit is more valuable. Anthropic's own glossary recommends ~0.0 for analytical work.

**Source:** doc 14 Section D.

---

## 37. Formulary not-found message rewrite

**Decision:** Replace "Use your clinical training knowledge for dosing" (the documented historical failure-mode anti-pattern) with a structured error payload telling the AI to add the drug to `omitted_medicines[]` with `reason='not_in_formulary'`. The server enforces this; the AI cannot bypass.

**Rationale:** The current message actively invites the failure mode it should prevent. Rewriting it to a structured instruction, enforced server-side, makes the safe path the only path.

**Source:** doc 11 CRITICAL-2; doc 14 Section B.
